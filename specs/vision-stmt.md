# Vision

This is the vision statement for a digital simulator with an audience
of retro computing hobbyists who design CPUs with old-school TTL
components.

The simulator is composed of first, a design canvas in the web
browser. The canvas has a tool pallet of TTL components and the
work surface on which a design can be drawn.

After placing TTL components on the design canvas, the user can
wire them together by drawing lines. Lines include buses of specified
widths. Some UI features are described in more detail later. Other
features must be fleshed out during discussion.

This web part of the design editor is a localhost-only application
that communicates with a small localhost-only Web server capable
of accepting the design over the server connection and saving it
to a local file. The small server will eventually offer additional
API calls so it must have a well-designed, expandable API.  Designs
are saved in a custom format. Most retro computing hobbyists do not
prefer Verilog, so this software will not use it.

The name, metadata, and behavior of each TTL component is specified
by a small textual file (MD file). The format of the file is to be
determined but the functionality of each TTL component should be
represented in
[GALasm](https://forum.vcfed.org/index.php?threads/galasm-manual.1242876/).
([examples](https://github.com/daveho/GALasm/tree/master/examples)).

When the user is satisfied with the design, they can press a button
which causes a C program to be generated by the server. The generated
C program is a simulator for the design. It incorporates the logic
equations for each component translated from GALasm to C.

Thus the server must implement an API for the drawing application
and a transpiler to create C-language simulation engines on demand.



## User Interface

In my opinion, no ideal user interface for digital circuit design
has ever been created. The difficulties surround what happens to
the wires, particularly when components are moved on the drawing
surface by user actions. Fully automatic wire routing is too difficult
and does not result in readable designs.

My preferred answer is the answer used in the software application
called "Fritzing". The entire display surface is backed by a fine
grid, roughly 1mm or 2mm screen spacing. Components and wires, and
their wire endpoints ("anchor points"), are all located on the grid.
When a wire is created between two component anchor points, it is
immediately drawn in as a straight line. This would form a rat's
nest if many search wires were drawn. The user can then click on
any point in the wire to create a "bend poimt" and move then drag
that point to any desired location on the grid, rubber-banding the
two parts of the wire that touch the moveable bend point. The user
can then proceed to add as many bends as desired, with each click
dividing the clicked segment into two. It is anticipated that the
user will generally form Manhattan-routed wires between anchor
points on components.

When a component is moved, the straight line segments closest to the
component are stretched even if doing so causes them to cross another
component. After such a move action, the user can click the stretched
line segment repeatedly, dividing and rubber-banding it to form a Manhattan
routed wire around any intervening components. The user can also
click an existing bend point and drag it or right click on a bend
point that has been introduced into a wire and select "delete bend
point". The wire then becomes a straight run between the nearest
surrounding end points or bend points.

## Abstraction level

As a TTL simulator, this design must be capable of representing
three-state buses and high impedance output states. Therefore, it
will implement what is sometimes called "four-level" logic where
the four states are 1, 0, undefined, and high impedance.  In this
simulation there is no reason to represent power or ground, or to
represent capacitors or any analog electronic behaviors.

With respect to timing, component meta-data may optionally include
delay timings in nanoseconds. In this case transpilation of the
design can, at the users option, include dynamic delay computation.
If delay times are not given or the user does not specify their
use, the Simulator uses what is called a unit delay model. In the
unit delay model, each layer of combinational logic requires one
unit of time. This at least imposes in ordering on events without
properly simulating propagation delay.

## Eventual Toolset

I envision additional tools eventually, for example the translation
of the save file format to a basic KiCad schematic, and a tool to
allow the easy definition of new components. But all such discussions
are deferred for now.

## Architecture

The user interface in the browser and the server must of course be
separate components. Please use JavaScript for the user interface
and golang for the server. The transpiler should be a separate
command line program so that it can eventually participate in a
pipeline with the C compiler to generate an executable simulator
for the design. Although the transpiler may be written in any
language, I prefer Go because I am already familiar with it. Please
use this language unless there is a strong engineering reason to
prefer a different one in which case let's discuss.
