# Requirements: TTL Circuit Design Editor

> This document, together with `design.md`, is the single source of truth for
> the system's intended state and is kept current. `CHANGELOG.md` is a
> chronological index of change requests (history and rationale only) — it is
> not needed to determine current behavior.

## 1. Overview

A localhost-only digital circuit design editor for retro computing hobbyists who design CPUs and other digital hardware using classic TTL components. The system consists of a JavaScript web application running in the browser and a small Go server running on the same machine. The browser application provides a schematic-style canvas on which the user places and wires TTL components. The server stores and retrieves designs and hosts the component library. Both simulation engines described in the vision statement (`sim-vision.md`) are now in scope: the slow ("debug") simulator (§3.19) and the fast engine — a code generator that produces a standalone C simulation program (§3.23). (Supersedes the original "simulation out of scope for this phase" and the later "fast engine out of scope".)

---

## 2. Users and Roles

**Designer (the only user role):** A single local user. There is no authentication, multi-user access, or network exposure. The server listens on localhost only.

## 3. Functional Requirements

### 3.1 Application Shell and Startup

- FR-001: The system shall consist of a JavaScript single-page application served by a Go HTTP server bound exclusively to `localhost`.
- FR-002: On startup, the server shall load all component definition files (YAML files) from a configured component library directory.
- FR-003: The browser application shall retrieve the component library from the server at startup and populate the palette before allowing the user to interact with the canvas.
- FR-004: The application shall open in select-tool mode with an empty, unsaved design named `"unnamed schematic <datetime>"` where `<datetime>` is the current local date and time.
- FR-004a: The application's top chrome shall be a single horizontal bar combining a **menu bar** and **modal-tool buttons**. Its left portion holds pull-down menus and its right portion holds always-visible buttons. The menus are: **File** — New (FR-044), Open (FR-052), Save (FR-046), Save As (FR-049), Export… (FR-119), Refresh Types (FR-088); **Edit** — Undo, Redo (FR-024), Copy, Paste (FR-111/FR-112); **View** — Zoom In, Zoom Out (FR-022), Fit to Screen (FR-022a); **Simulate** — Test Vectors… (FR-115; toggles the docked test-vector panel open/closed, FR-115b, which imposes a read-only lock — including disabling the Run/Stop button — while open, FR-115h) and Generate C… (FR-116). The buttons are: the modal tool buttons **Select**, **Wire** (FR-026), and **Bus** (FR-035) — exactly one is active at any time and the active one is highlighted (the Wire button shows the wire-cursor icon, FR-025) — followed by the **Run/Stop** button (FR-076). A menu opens on click; choosing an item performs its action and closes the menu; the menu is also dismissed by clicking outside it or pressing Escape. While a simulation runs, the design-modifying commands are disabled exactly as before (FR-087): the Wire and Bus buttons, the Undo, Redo, Paste (FR-112), New, Open, and Refresh Types items, the Export… item (FR-119), the Generate C… item (FR-116), and the Test Vectors item (FR-115b) are disabled, while Save, Save As, the zoom items, Select, and Run/Stop stay enabled. This combined bar supersedes the former flat toolbar — a single row of text/icon buttons — every one of whose buttons now lives either in a menu or as one of the retained modal-tool/Run buttons; it was adopted because the growing command set no longer fit as a row of buttons. (Added 2026-06-21; menu inventory and lock lists brought current 2026-07-08 — Export…, FR-119, and Generate C…, FR-116, added; Paste, FR-112, included in the simulation-disabled list; Save As confirmed enabled during a run.)
- FR-004b: Every menu item (FR-004a) that has a standard keyboard accelerator shall display that accelerator right-aligned within its menu row, and the editor shall bind it globally through the same key handler as the existing shortcuts — ignored while a text field is focused and honoring the simulation lock (FR-087) and per-item enablement. The accelerators are: **File** — Open `Ctrl/Cmd+O`, Save `Ctrl/Cmd+S`, Save As `Shift+Ctrl/Cmd+S` (New and Refresh Types have **none**: `Ctrl/Cmd+N` is reserved by the browser and cannot be intercepted, and Refresh Types has no standard accelerator); **Edit** — Undo `Ctrl/Cmd+Z`, Redo `Shift+Ctrl/Cmd+Z`, Copy `Ctrl/Cmd+C`, Paste `Ctrl/Cmd+V` (already bound per FR-024/FR-111/FR-112, now also shown in the menu); **View** — Zoom In `Ctrl/Cmd+=` (also `Ctrl/Cmd++`), Zoom Out `Ctrl/Cmd+-` (FR-022) (Fit to Screen, FR-022a, has **none**). The newly added File and View accelerators suppress the browser default (page save / open dialog / browser zoom). The hint text is platform-appropriate (⌘/⇧ symbols on macOS, `Ctrl+`/`Shift+` elsewhere). (Added 2026-06-25.)
- FR-004c: The editor shall additionally provide **canvas key bindings that have no menu item**, bound through the same global key handler as the accelerators (FR-004b) — ignored while a text field is focused and honoring the simulation and test-vector locks (FR-087/FR-115h) exactly as the commands they invoke do: **`w`** activates the Wire tool and **`b`** the Bus tool (FR-026/FR-035, from select mode); **`r`** rotates the selection 90° clockwise and **`Shift+r`** counter-clockwise (FR-019); **`Delete`/`Backspace`** delete the selection (FR-018a/FR-033d; while a conductor is in progress, `Backspace` instead removes the last locked waypoint, FR-027e, and `Esc` cancels); with a **single bus or bus-segment selection**, **`+`/`-`** (also `=`) increase/decrease that bus's width (minimum 1), an alternative to the context menu's Set width… (FR-038); and **`Ctrl/Cmd+Y`** is an additional Redo alias alongside FR-004b's `Shift+Ctrl/Cmd+Z`. (Added 2026-07-08: these bindings had long existed in the editor and the user manual but were never specified — recorded here so the specs remain the source of truth.)

### 3.2 Component Palette

- FR-005: The palette shall display one fixed-size tile per loaded component type. Each tile shall show the component type's full external **display name** as its button text — the unabbreviated type name for a 74-series part (e.g. "74138", "7400") and the part number for a GAL part (FR-005b) — rendered at a font size small enough that a five-character name (e.g. "74125") fits within the fixed tile. The display name is also presented in the tile's tooltip (FR-005a). (Reworked 2026-06-22; supersedes the prior rule that stripped the leading "74" to a 2–3 digit abbreviation — the tile now shows the full, free-form display name, distinct from the type's internal id, FR-066e/FR-005b.)
- FR-005a: When a component type declares a one-line description (FR-104), the tile's tooltip shall also present it, so hovering a palette tile shows both the full display name and what the part does (e.g. "74138: 3-to-8 line decoder/demultiplexer, inverting"). A type with no description keeps the plain display-name tooltip of FR-005.
- FR-005b: A GAL part (a component type declaring `gal:`, FR-066a) is identified for display by its `partnumber` (FR-066b): its tile shall be labeled with the part number as its button text (FR-005), and its tooltip shall lead with the part number, then the one-line description (FR-005a) when present (e.g. "PC-DECODE-A — program-counter address decode"). The device family (the `type`, e.g. "22V10") is no longer the tile label. (Reworked 2026-06-22; supersedes the earlier rule that labeled a GAL tile with its device family and showed the part number only in the tooltip — the part number is now the visible button text, consistent with FR-005's full-name labeling. The part number is a free-form display name, not the library key, FR-066e.)
- FR-006: The palette shall arrange tiles in a fixed-width grid of equal-size tiles (3 per row), packed left-to-right then top-to-bottom. Upper-region order: 74-series parts first, ascending by numeric part number (the digits after the "74" family prefix), followed by the free-form-named parts (GAL parts, FR-005b, and generated memory devices, FR-114c); ties, and the free-form group itself, are ordered by library id (FR-066e). (Reworked 2026-07-08; supersedes "ascending order of the numeric abbreviated part number", which predated the FR-005 full-display-name rework and defined no order for free-form-named parts. Supersedes the prior flat, unordered list.)
- FR-006a: The palette shall be divided into two equal-height regions by a horizontal divider at its vertical midpoint. Each region shall carry a small fixed heading bar at its top, above its tiles, which does not scroll with the tiles and uses the same small font as the tile labels: the upper region's reads "TTL Components" and the lower region's reads "Built-In Components". The upper region shall hold the 74-series component tiles (FR-005, FR-006). The lower region shall hold built-in editor objects (FR-067). Each region's tiles shall scroll independently (beneath its heading) when its contents overflow. The lower-region tiles follow the same grid and click/drag-to-place behavior as the upper region (FR-006, FR-008, FR-009, FR-009a) but are labeled by a distinct icon and tooltip per object rather than a part-number display name (FR-005). (Heading bars added 2026-06-27.)
- FR-007: The component library shall be loaded once at server startup; the server is not required to detect or reload YAML files added while running. (Exception: parts authored in-app are created and made live without restart per FR-007a.)
- FR-007a: The application shall provide a server endpoint to **create** a component definition while running: it accepts an authored part — a GAL part (FR-066c) **or** a generated memory device (FR-114/FR-114c) — writes its YAML into the component library directory under a filename derived from the type's `id` (FR-066e), and adds it to the running server's in-memory library so subsequent `/api/v1/components` responses include it. A submission must be identifiable as authored: it carries either a `partnumber` (a GAL part, FR-066b) or a `mem` block (a memory device, FR-114c); the server rejects one with neither. **Creation only, never overwrite:** the server refuses a create whose `id` collides with an existing library type, or whose derived filename already names an existing file — it never overwrites or edits an existing library file from the app (a full in-app editing interface is a future change). After a successful create the client shall add the new tile to the palette **live** (no restart or page reload), in the upper region (FR-006a) alongside the other loaded parts. This is a deliberate, additive exception to FR-007's startup-only load and to FR-088's "restart to pick up YAML edits"; it covers only in-app *creation* of a new part and does not add general filesystem watching or edit-reload of externally changed files.

### 3.3 Component Placement

- FR-008: The user shall be able to place a component by dragging its tile from the palette onto the canvas.
- FR-009: The user shall be able to place a component by clicking its tile in the palette and then clicking a point on the canvas.
- FR-009a: While a palette tile is armed for click-to-place (FR-009), that tile shall show a pressed-in (inset) appearance distinguishing it from the unarmed, raised tiles, and shall return to the raised appearance once placement completes or is cancelled.
- FR-010: In both cases, placement shall be one-shot: after the component is placed the application shall return to select-tool mode automatically.
- FR-011: On placement, the system shall assign the component a **reference designator** (`refdes`: U1, U2, U3, …) incremented from the highest existing U-designator in the design. The `refdes` is the component's **immutable internal identity** — what wiring, bus group-snaps, selection, persistence, and every other reference use to refer to the instance — and is unique within the design; it is auto-allocated and not user-editable. The *displayed* designator is a separate, free-form **label** (FR-011b) that defaults to the `refdes` and may be edited or duplicated freely without affecting identity. A subunit-rendered package (FR-013a) consumes a single U-number shared by all of its subunits, whose `refdes` append a letter suffix in unit order (e.g., U5A, U5B, U5C, U5D). (Reworked 2026-06-22; the displayed designator was split from the internal identity so it can be edited freely — the `refdes` remains the stable identity, and a `label` carries the editable text, FR-011b.)
- FR-011a: Built-in editor objects (FR-067) shall instead be assigned a `refdes` from a separate series (A-1, A-2, A-3, …), incremented from the highest existing A-number in the design, so they do not consume IC U-numbers (FR-011). As with ICs, the `refdes` is the internal identity and the displayed designator is the editable `label` (FR-011b) defaulting to it. Exception: the text note built-in (FR-071f) is a pure annotation; its `refdes` (from a separate `N-<n>` series) is internal-only and never displayed, and it has no editable label. (Reworked 2026-06-22.)
- FR-011b: Each component instance shall carry a free-form display **label** for its reference designator, defaulting to the `refdes` (FR-011). The user shall be able to edit this label to any text through the properties panel (FR-020a), as an undoable action (FR-024), with **no** sanity checks: it need not match the U/A series format and may duplicate another instance's label. Because the label is display-only and the instance is identified internally by its immutable `refdes` (FR-011), editing or duplicating it does not affect wiring, bus group-snaps, the netlist, or any other reference. Clearing the label restores the default (the `refdes`). The text note (FR-071f) shows no designator and so offers no such field. (Added 2026-06-22.)
- FR-012: Each component instance shall display its reference-designator **label** (FR-011b, e.g. "U3" by default) and its type's display name (e.g., "74138"; the part number for a GAL, FR-005b) as text labels on the canvas. These labels shall always render upright regardless of the component's rotation. Exception: the text note built-in (FR-071f) draws no designator or type-name label — only its (rotatable) text, plus a dotted outline box when selected or editing.
- FR-012a: Component labels shall be progressively culled as the symbol shrinks on screen, so dense text does not overwhelm a small symbol. Label text is drawn at a fixed pixel size while the symbol scales with zoom; the governing quantity is the symbol's apparent on-screen size, taken as its smaller footprint dimension times the current pixels-per-grid-unit scale. At two descending thresholds (T1 > T2): below **T1**, pin name labels (FR-015) are no longer drawn; below **T2**, the type display-name label is no longer drawn either, leaving only the reference-designator label (the U-number), which is always drawn (and is centered on the symbol once its type label is suppressed). Above T1 all labels render as before. For subunit-rendered components (FR-013a) only the T1 pin-label rule applies (they show no type-name line); built-ins (which draw no pin labels and no type-name line) are unaffected. The thresholds are fixed pixel constants chosen for legibility, so a physically larger symbol keeps its labels to a lower zoom than a small one at the same zoom. (Added 2026-06-27.)

### 3.4 Component Appearance

- FR-013: Each component shall be rendered as a rectangular outline with a short connection lead at each pin and pin name labels on the sides of the rectangle. Each lead shall be drawn as a straight line segment emanating from the pin's grid point on the outline edge, outward along the pin's side direction (rotation-aware), a length of 0.5 grid units. The grid point remains the wire-connection coordinate. Leads are short enough that adjacent pins (one grid unit apart) do not crowd. The drawn wire attachment point and the wire hot region are specified by FR-013d. This lead treatment applies to every non-subunit component, built-in objects included (their connection pins carry leads; their own body glyphs are unaffected); subunit-rendered components (FR-013a) mark pin connection points per FR-013c instead, so the circle stays reserved for logic negation. (Reworked 2026-07-06; supersedes the tangent connection bubble/circle — a straight outward lead replaces it. Earlier reworked 2026-06-11 from "clicking anywhere within a pin's bubble starts or ends a wire" — see FR-013d.)
- FR-013a: A component whose YAML declares `rendertype: subunit` shall be rendered as a set of separate traditional schematic symbols — one per functional unit — rather than a single rectangle (FR-013). On placement, all of the package's units shall be dropped onto the canvas at once, offset so they do not overlap, each independently selectable, movable (FR-017), and rotatable (FR-019).
- FR-013b: The renderer shall provide the schematic symbols named by `renderas`: `nand`, `and`, `or`, `nor`, `xor`, `xnor`, `not`, `mux2`, `mux4`, `mux8`. Gate symbols draw their inputs on the left and their output on the right; the number of inputs is the count of the unit's input pins. Multiplexers draw as a symmetrical trapezoid whose long (left) side carries the data inputs and whose short (right) side carries the output, with the top and bottom edges sloping about 30° toward the right; select inputs enter the top. Every connection point — including multiplexer selects — shall lie on a grid intersection (FR-021); where the symbol edge does not pass through that intersection (the sloped mux top), a short stub shall connect the on-grid connection point to the edge, and the grid intersection shall remain the wire-connection coordinate (FR-013). Connection points are marked per FR-013c.
- FR-013c: For subunit-rendered components (FR-013a), a pin's connection point shall not carry a resting bubble (so the circle remains exclusively a logic-negation indicator, and a positive-true pin is not mistaken for an inverted one). Instead, when the subunit symbol is hovered or selected, each connection point shall be marked with a short tick drawn as an outward lead along the pin's direction from the pin's grid point (so it reads as a connection stub rather than disappearing into a body edge); at rest no such mark is shown. An inverting output is exempt: its negation bubble already marks the connection point, so it receives neither a resting bubble nor a hover/select tick. The pin's grid point remains the wire-connection coordinate (FR-013) whether or not a tick is shown.
- FR-013d: Wires and buses shall be *drawn* attaching at a pin's visual attachment point: the outer end of the pin's lead for pins that carry one (FR-013, built-ins included) — the grid point plus 0.5 grid units outward along the pin's side direction (rotation-aware) — or the pin's on-grid connection point for subunit-rendered pins (FR-013c; an inverting output's wire still attaches outside its negation bubble, never inside it). This is rendering only: the pin's grid point remains the electrical wire-connection coordinate and the persisted endpoint (FR-021, FR-059). A pin's hot region — the area over which the cursor becomes the wire cursor (FR-025, FR-027b) and a click starts or terminates a wire — shall be a circle of radius approximately 0.7 grid units centered on the visual attachment point; where the hot regions of nearby pins overlap, the nearest pin shall win. (Reworked 2026-07-06: the attachment point moved from the bubble center, 0.25 out, to the lead's outer end, 0.5 out, when the bubble became a lead per FR-013. Earlier reworked from FR-013's "the bubble is the click target"; the prior hot region was a 0.5-grid-unit circle on the grid point.)
- FR-014: The position and side (left, right, top, bottom) of each pin shall be determined by the component's YAML file; the editor shall not infer or rearrange pin positions automatically.
- FR-014a: For subunit-rendered components (FR-013a) the position of each pin is dictated by its symbol, not by the YAML; such pins specify their `unit` (a letter) instead of a position, and their order within a unit in the YAML determines slot order on the symbol (inputs top-to-bottom; multiplexer selects in least-significant-first order).
- FR-015: Pin name labels shall always render upright regardless of the component's rotation.
- FR-015a: A unit-rendered component's (FR-013) pin name labels shall be placed so that each name hugs its pin's border edge, separated from that edge by a small fixed pixel margin, and grows *inward* (toward the body center) as the name lengthens. A left-edge pin's label is left-aligned just inside the left border; a right-edge pin's is right-aligned just inside the right border; top- and bottom-edge pins are aligned analogously against their edge. This guarantees a constant, legible gap between every pin name and the component border regardless of name length, without pushing short names toward the center. The labels remain upright (FR-015) and fixed-pixel with the same zoom culling (FR-012a). (Added 2026-07-05; supersedes the earlier fixed inward nudge that centered each pin name a constant offset inside the border — which, being symmetric about that point, let a longer name's near edge crowd or cross the border line and become hard to read. Subunit-rendered pin labels, which hang from the symbol outline via `pinLabelEdge` to avoid stubs, are unaffected.)

### 3.4a Built-in Objects

- FR-067: The editor shall provide a set of built-in objects — entities with functionality built into the editor and eventual simulator, as distinct from 74-series parts loaded from YAML. Built-in objects are defined by the client application (not by YAML files), placed from the lower palette region (FR-006a), and once placed behave as component instances: selectable, movable, rotatable, deletable, persisted in the design, and wireable through their connection points. They are designated A-1, A-2, … (FR-011a).
- FR-067a: Each built-in object shall be associated with a behavior: a simulation-semantics function implemented in client JavaScript and registered alongside the object's type definition (it is the built-in analogue of a 74-series part's GALasm equations, FR-066). The behaviors' semantics are specified in §3.19: the clock per FR-084, pull-up/pull-down per FR-083, the indicator per FR-068 (display only; it drives nothing), the power-on reset per FR-071b, the input switch per FR-071c/FR-087a. Behaviors are code, not data: they are resolved from the client registry by type name and are never serialized into saved designs. Exception: the text note built-in (FR-071f) is a pure annotation with no pins and no behavior — it has no entry in the behavior registry and does not participate in simulation. (Reworked 2026-06-11; supersedes "interface deferred to the simulator design".)
- FR-068: The first built-in object shall be a state indicator. It occupies a 2×2 grid-unit footprint and exposes a single input connection point on the bottom edge (centered) that may be wired to any wire or driven output. It is drawn as a round bubble sized to fit comfortably within the square. The indicator is not independently stateful: it always displays the state of the wire to which it is attached. It has three displayed states: undriven/undefined — a medium-gray bubble with a black "?"; logic 1 — a white bubble with a black "1"; logic 0 — a black bubble with a white "0". While a simulation runs (§3.19) the indicator displays its net's simulated value, with both U and Z (FR-077) rendered as the undriven "?" state; when no simulation has run it displays the undriven state. After a simulation terminates, the last simulated values remain displayed until the design is next modified (FR-085). The same bubble image is used for the palette icon (FR-006a) and the placed object. (Reworked 2026-06-11; supersedes "always displays the undriven state until the simulator exists".)
- FR-069: A built-in logical pull-up object. It occupies a 2×2 grid-unit footprint and exposes a single connection point on the bottom edge (centered). It is drawn as a two-headed upward arrow: two upward-pointing chevrons (carats) stacked at the top of the symbol, with a vertical line bisecting the symbol that rises from the bottom connection point up to just below the chevrons without intersecting them. Its palette tooltip is "pull up". The same image is used for the palette icon and the placed object.
- FR-070: A built-in logical pull-down object. It occupies a 2×2 grid-unit footprint and exposes a single connection point on the top edge (centered). It is drawn as an upside-down "T": a long vertical stem descending from the top connection point to a short horizontal bar near the bottom. Its palette tooltip is "pull down". The same image is used for the palette icon and the placed object.
- FR-071: A built-in clock generator object. It is drawn as a box containing the letters "CLK" and exposes a single connection point on the right edge (centered). Its palette tooltip is "clock". The same image is used for the palette icon and the placed object.
- FR-071a: The clock generator shall declare two properties (FR-020b): `period` — the simulated clock period in nanoseconds (default 100) — and `speed` — the human-perceived clock rate in hertz, i.e. cycles of the simulated clock per real second (default 1). The future simulator shall therefore advance period × speed nanoseconds of simulated time per real second: setting both to 1 simulates an unrealistically fast 1 ns per second; the defaults give 100 simulated ns per real second. The other built-ins (FR-068–FR-070) declare no properties.
- FR-071b: A built-in power-on reset object. It is drawn as a box containing the letters "RST" and exposes two connection points on the right edge: R (active high) and /R (active low). Its palette tooltip is "power-on reset"; the same image is used for the palette icon and the placed object. It declares one property (FR-020b): `cycles` — the reset duration in clock cycles (default 3). From simulation start, R shall drive 1 and /R shall drive 0 for the first cycles × period simulated nanoseconds, then R drives 0 and /R drives 1 for the remainder of the run. `period` is the effective period (FR-071a) of the design's clock generator when the design contains exactly one; otherwise (no clock, or several) the 100 ns default. With the FR-084 waveform (first rising edge half a period in) the reset thus spans the first `cycles` rising clock edges, releasing half a period after the last.
- FR-071c: A built-in interactive input switch object. It occupies a 2×2 grid-unit footprint and exposes a single output connection point on the right edge (centered). It has two states, 1 and 0. It is drawn like the state indicator (FR-068) — a round bubble showing its value (a white bubble with a black "1", or a black bubble with a white "0") — but with a small arrow off the bubble toward the output pin marking it as a signal source (unlike the indicator, it drives a value and shows it whether or not a simulation is running). Its two states correspond to the simulator's logic values 1 and 0; it can no longer be set to the undefined value U. Its palette tooltip is "input switch"; the same image is used for the palette icon and the placed object. The current state is per-instance state, persisted in the saved design and restored on load; a newly placed switch defaults to 0. The switch declares no numeric properties (FR-020b). Its state may be set two ways: through the properties panel while editing (FR-020c) and, while a simulation runs, by clicking the switch (FR-087a). Its driving behavior is specified in §3.19 (FR-087a). (Reworked 2026-06-17; supersedes the three-position rotary dial — marks 1/0/? with a U state and default ? — whose marks were too small to read and whose U state added little value.)
- FR-071d: A built-in 8-wide state indicator object. It occupies a 3×9 grid-unit footprint and exposes eight input connection points `D0`–`D7` down its left edge, one per bit. The eight pins form a single pin group named `D` so that an 8-bit bus snap-connects to all of them at once (FR-041/FR-042), the bus adopting `D0`–`D7` as its bit names. It is drawn as an LED bar-graph: a rectangle divided into eight horizontal stripes, each stripe displaying the state of its bit exactly as the 1-wide indicator's bubble does (FR-068) — white for logic 1, black for logic 0, medium-gray for undriven/undefined (U and Z) — tracking the live simulated values during a run and retaining the last values after it ends until the design is next modified (FR-085). Like the 1-wide indicator it is display-only and drives nothing (§3.19). Its palette tooltip is "state indicator (8-wide)"; the palette icon is a bar-graph glyph (a rectangle of stripes). It declares no properties (FR-020b).
- FR-071e: A built-in **multi-bit port** / off-sheet connector object (`portN`), the N-bit analogue of the 1-wide port (FR-094). Its **width** N is chosen at placement: dropping a `portN` tile opens a modal dialog prompting for a bit width in the range **2–16** (default 8); cancelling places nothing. The chosen width is fixed for the life of the instance — it is **not** editable afterward (to change it, delete and re-place) — and there is no width control in the properties panel; the label is editable there like the 1-wide port's. The instance occupies a `3 × (N+1)` grid-unit footprint and exposes N connection points `P0`–`P(N-1)` down its left edge, one per bit (the on-sheet/"internal" connections). The N pins form a single pin group named `P` so that an N-bit bus snap-connects to all of them at once (FR-041/FR-042), the bus adopting `P0`–`P(N-1)` as its bit names. It is drawn on the canvas as N narrow pentagons, one roughly aligned with each bit pin (rows 1–N) and pointing off-sheet (apex away from the pins, like the 1-wide port, FR-094b); the pentagons carry no letter. The palette icon shows a shorter suggestive stack of the same flag shape — the placed object and the icon need not match exactly. Its palette tooltip is "port / off-sheet connector (multi-bit)". Like the 1-wide port it is a real interface element: it has a user-set signal **label** (defaulting to its refdes) and contributes to the design's external interface (FR-095) as a **pin group of N one-bit pins** (`<label>0`..`<label>(N-1)`, group named `<label>`) to which a matching-width bus snap-connects (FR-041/FR-042), with a **derived direction** (FR-094c) — one in/out/bidir value applied to the whole group, aggregated across its bits (bidir if any bit's net touches a bidirectional/three-state pin, else output if any bit is driven by a plain output pin, else input), to which a direction **override** (FR-094d) may apply when that aggregate is bidir. Unlike the 1-wide port — whose single connection point is a `connector` vertex joined to its net by label — a `portN`'s N pins join nets as ordinary pin members through the bus/wire snapped to them; consequently same-label per-bit net joining (FR-094a) and off-sheet cross-file links (FR-101) remain **deferred** for `portN` (it joins only through drawn buses/wires for now). It declares no numeric properties (FR-020b). (Reworked 2026-06-27; supersedes the fixed 8-wide, drives-nothing "grouped bus terminal only" form — the old `port8` type id `type-port8` is no longer recognized, a deliberate clean break — which never participated in the interface and so rendered nothing when its design was embedded.)
- FR-071f: A built-in text note object. Its palette tile is a box reading "NOTE" (like the clock/reset tiles, FR-071/FR-071b). On the canvas it is drawn as free-form text; a blue dotted outline box is shown around the text only while the note is selected or being edited, and at rest (unselected and not editing) only the text is drawn, with no outline. It is a pure annotation: it has no connection points or pins, does not participate in wiring, the netlist, or simulation, and declares no properties (FR-020b) and no behavior (FR-067a). It is exempt from the built-in A-N designator series (FR-011a) — it carries an internal-only `N-<n>` `refdes` (its identity, FR-011) that is never shown and has no editable label — and from the designator/type-name canvas labels (FR-012); only the dotted box and its text are drawn. The box shall auto-size to fit its text, snapped to whole grid units, from a small minimum. Text entry: on placement the note shall immediately enter text-entry mode; double-clicking a placed note shall re-enter it. The Enter/Return key shall terminate text entry (committing the text); Shift+Return shall insert a line break. The note shall be selectable, movable, deletable, and rotatable like other objects, including as part of a multi-object selection (FR-016a/FR-019); rotating it turns the text (and the outline box when shown) together as a rigid body about the rotation pivot, so the text reads at the rotated orientation — a deliberate exception to the upright-label convention of FR-012/FR-015/FR-020, which governs refdes and pin labels, not note content. Its text is per-instance state persisted in the saved design and restored on load; a newly placed note's text is empty. (Added 2026-06-22.)
- FR-071g: A built-in **transmission gate** object — an ideal bidirectional switch. It occupies a 2×2 grid-unit footprint and exposes three connection points: two contact terminals **A** (left edge, centered) and **B** (right edge, centered), and an active-high enable **EN** (top edge, centered). It is drawn as the conventional transmission-gate glyph — two overlapping triangles pointing in opposite directions between A and B — with the EN lead entering the top; the same image is used for the palette icon and the placed object, and its palette tooltip is "transmission gate". The two terminals are fully symmetric and interchangeable: neither is an input or an output, and drivers on either side may change arbitrarily. When the gate is **closed** (EN reads logic 1) its two terminal nets are electrically joined — merged into one net for resolution — and when **open** (EN reads 0) they are independent; the precise semantics, including the EN-reads-U rule, the one-unit control-to-contact delay, and the no-charge-storage non-goal, are specified by FR-083a. The gate itself drives nothing and stores nothing. It declares no properties (FR-020b) and is not an interactive input (FR-087b): while a simulation runs a click on it reports the selection lock (FR-087). In all other respects it is an ordinary built-in (FR-067): A-series refdes (FR-011a), lower palette region (FR-006a), selectable/movable/rotatable/deletable/persisted/wireable. (Added 2026-07-07.)
- FR-071h: A built-in **relay** object with a single-pole double-throw (changeover) contact. It occupies a 4×4 grid-unit footprint and exposes four connection points: a coil control input **COIL** on the top edge and three contact terminals on the right edge — **NO** (normally open, top), **COM** (common, middle), and **NC** (normally closed, bottom). The coil is idealized as a single logic-level input pin — no second coil terminal and no coil current or voltage, consistent with the system's no-analog scope: the relay is **energized** when COIL reads logic 1 and **released** when it reads 0. When released, COM and NC are joined and NO is isolated; when energized, COM and NO are joined and NC is isolated — in each case the joined terminals' nets merge into one net for resolution per FR-083a, which also specifies the COIL-reads-U rule and the one-unit delay from a coil change to the contact change. There is **no pick/drop (operate/release) time** in this version: the contacts follow the coil with the standard one unit (FR-078); a numeric delay property (FR-020b) is a possible later additive change. An SPST contact is obtained by leaving the unused throw (NO or NC) unwired. It is drawn as a schematic relay — a coil whose single logic-level lead enters from the top, and on the right the three contact terminals: COM (the common pole, marked with a dot) plus NO and NC. **No moving contact arm is drawn**: it could not track the simulated contact state, so a static one (fixed in some position) misleads more than it helps — the live state is read from wired indicators. The three right-edge terminals are labeled **NO**/**COM**/**NC** on the canvas (in a column between the coil and the contact) so they can be told apart; these labels are culled at low zoom like other pin names (FR-012a). The glyph without labels also serves as the palette icon, and its palette tooltip is "relay (SPDT)". It declares no properties, is not an interactive input (FR-087b), and is otherwise an ordinary built-in exactly as the transmission gate (FR-071g). (Added 2026-07-07.)

### 3.5 Component Selection and Movement

- FR-016: In select-tool mode, the user shall be able to click a component to select it.
- FR-016a: In select-tool mode the selection may contain multiple objects of any mix of kinds. Shift-clicking a selectable object (component, wire, or bus) shall toggle its membership in the selection — adding it if absent, removing it if present. A plain (non-shift) click shall replace the selection with the single clicked object (FR-016); a left click on empty canvas shall clear the entire selection (FR-023a). Operations act on every selected object: Delete removes all selected objects (FR-018a); dragging any selected component moves all selected components by the same grid offset (FR-017), carrying the wiring interior to the selection with them (FR-018c); Rotate turns the whole selection as a single rigid group about one pivot (FR-019), carrying interior bends and junctions with it (FR-018c). Selected wires and buses are not themselves moved or rotated as standalone objects, but endpoints connected to a moved or rotated component follow their pins, and bends and junctions interior to the moving or rotating selection travel with it (FR-018/FR-018c/FR-019).
- FR-016b: In select-tool mode, pressing and dragging the left mouse button on bare canvas shall draw a rubber-band selection rectangle (FR-023a). The rectangle's mode is determined by the horizontal drag direction: when the release point is to the right of the press point it is a *window* selection — only objects (components, wires, and buses) lying entirely within the rectangle are selected; when the release point is to the left of the press point it is a *crossing* selection — every object the rectangle touches (intersects or encloses) is selected. The vertical direction is irrelevant. A plain rubber-band replaces the selection; holding Shift adds the rectangle's hits to the existing selection. The selection updates live as the rectangle is dragged, and the rectangle is rendered with a visually distinct appearance for window versus crossing mode. Pressing Escape during the drag cancels it and restores the pre-drag selection. A press released without dragging is a click, not a rubber-band (FR-016a/FR-023a). The rubber-band selects only whole top-level objects (a touched/enclosed wire or bus is selected in its entirety, not by segment, FR-031); pins, bend points, junctions, and individual segments are not independently rubber-band-selectable.
- FR-017: The user shall be able to drag a selected component to a new position on the canvas; the component shall snap to the grid. (Extended by FR-016a for multi-object selections.)
- FR-018: When a component is moved, any wire or bus segments directly connected to its pins shall stretch to follow, maintaining connectivity. A bus endpoint snap-connected to one of the component's pin groups (FR-041a/FR-042) counts as connected to that component and follows the move likewise, even though its endpoint vertex is `free` (its connection is the group binding, not pixel geometry). The stretched segments may cross other components; the user is responsible for re-routing them afterward. (See FR-018c: wiring interior to a multi-component move translates rigidly instead of stretching.)
- FR-018c: When components are moved together (FR-016a/FR-017), the wiring interior to the moving selection shall translate rigidly with them: for any wire or bus conductor network all of whose component connections are to components in the moving set, the network's bend points and its junction and free (dangling) vertices shall shift by the same offset, preserving the sub-circuit's shape. A *component connection* is a pin/connector endpoint **or** a group-snapped bus endpoint (FR-042); the latter is a `free` vertex but is bound to its component, so it counts toward this test and is shifted with the move just like a pin endpoint. A network with a component connection on a component outside the moving set is not interior and stretches per FR-018 — but a group-snapped bus endpoint on a *moving* component still follows that component (FR-018), so the bus stays attached and only stretches. (For a single moved component the usual case — wires running to other, unselected components — still stretches per FR-018; only wiring entirely self-contained on the moved component, such as a dangling stub, travels with it.) (Extended 2026-06-18; previously "pin endpoints" only, which left a group-snapped bus behind when its component moved.)
- FR-018a: In select-tool mode, the user shall be able to delete a selected component. Wires and buses connected to the deleted component shall remain in the design with their formerly-connected endpoints left dangling (see FR-029, FR-030). (Extended by FR-016a for multi-object selections.)
- FR-018b: Deleting any subunit of a subunit-rendered package (FR-013a) shall delete all subunits of that package as a single action; the system shall require explicit user confirmation (a warning dialog) before performing the deletion.

### 3.5a Copy and Paste

- FR-111: The user shall be able to **copy** the current selection to an in-application clipboard, via an Edit-menu **Copy** command (FR-004a) and the Ctrl/Cmd+C shortcut. Copy reads the selection and does not modify the design (it is not an undoable action). The clipboard captures a self-contained **fragment**: the selected components together with the wiring **interior** to the selection — a wire or bus conductor network all of whose component connections are to selected (copied) components, the same interior test as a rigid group move (FR-018c) — including those conductors' bend points, junction/free vertices, bus group connections (FR-042), and per-bit names (FR-037b). A conductor network with any connection to a non-copied component is excluded. Selecting any subunit of a subunit-rendered package (FR-013a) copies the whole package (all its siblings). Selected wires or buses that are not interior to the copied components are ignored; copy is component-centric. Copying an empty selection (or a selection with no components) is a no-op. The clipboard holds the most recent copy for the duration of the session — it survives New and Open (FR-044/FR-052) and is replaced by the next Copy — and is neither the operating-system clipboard nor saved with any design.
- FR-112: The user shall be able to **paste** the clipboard fragment into the current design, via an Edit-menu **Paste** command (FR-004a) and the Ctrl/Cmd+V shortcut. A paste is a single undoable action (FR-024). Pasting allocates a **fresh reference designator** (`refdes`, the internal identity, FR-011) for every pasted component — a U-number for an IC (FR-011) and an A-number for a built-in object (FR-011a), each incremented past the design's current maxima — so the pasted copies never collide with existing instances; a pasted subunit-rendered package (FR-013a) consumes one new U-number whose siblings carry the letter suffixes (FR-011). Pasted vertices, wires, and buses likewise receive fresh internal ids, and all interior connectivity is reproduced among the pasted objects (bus group connections and per-bit names included), independent of the originals. Per-instance type data, overrides (FR-058), input-switch state (FR-071c), and an edited display label (FR-011b) are carried into the copies; a copy with no explicit label keeps defaulting to its new `refdes`. A pasted port (FR-094) whose label is still its original refdes (the default) adopts its new refdes as its label, so it forms its own net; a port carrying a user-assigned label keeps that label. After a successful paste the pasted objects become the selection (FR-016a). Paste is disabled while simulating (FR-087) and is a no-op when the clipboard is empty.
- FR-113: Paste shall position the fragment **at the cursor**. On Paste the fragment becomes a floating, translucent preview that follows the pointer; the next left-click on the canvas **drops** it — snapped to the grid (FR-021) — committing the paste at that location, and the application returns to select-tool mode with the pasted objects selected (FR-112). Pressing Escape, or switching tools, cancels the pending paste without modifying the design. Only a uniform, whole-grid-unit translation is applied while placing; the fragment's internal geometry — relative positions, rotations, and bends — is preserved, and a chosen anchor point of the fragment tracks the cursor. A paste invoked from the Edit menu (when the pointer is not over the canvas) shows the preview at the canvas center until the pointer enters the canvas, then tracks it.

### 3.6 Component Rotation

- FR-019: The user shall be able to rotate the selection 90° clockwise or 90° counter-clockwise. The entire selection rotates as a single **rigid body** about one pivot: every selected component **and** every bend and junction interior to the selection (FR-018c) is rotated about that pivot, so the sub-circuit's shape and internal wiring are preserved — only its orientation and position change. The pivot is a grid intersection (FR-021): for a single selected component it is that component's placement origin (so a lone component still rotates exactly in place as before); for a multi-component selection it is the grid-snapped center of the selected components' combined bounding box. Wiring with an endpoint on a non-selected component is not interior and stretches per FR-018 (only its pin endpoints move). (Reworked 2026-06-16; supersedes "each selected component rotates about its own center" — under that rule a multi-component selection spun each part independently and left interior junctions and bends behind, tearing the sub-circuit apart. Extended by FR-016a for multi-object selections.)
- FR-020: Rotation shall reposition pin leads accordingly; all text labels (pin names, reference designator, type name) shall remain upright after rotation. Exception: a text note's content (FR-071f) is not such a label — it rotates with the note's box as a rigid body and therefore reads at the rotated orientation.

### 3.6a Per-Instance Type Overrides

- FR-020a: The user shall be able to view the type data of a selected component instance and override specific values (e.g., propagation delay) for that instance only. Overrides shall not affect other instances of the same type or the underlying YAML file, and shall be persisted per FR-058. The panel shall additionally present the instance's reference-designator label as an editable, free-form field (FR-011b). When the selection is empty or contains more than a single component, the properties panel is blank.
- FR-020b: A component type may declare named numeric properties, each with a name, a unit, and a default value (e.g., the clock's period in ns, FR-071a). Per-instance property values shall be viewable and settable through the same per-instance override mechanism as other type data (FR-020a) and persisted per FR-058. Built-in objects (FR-067) declare their properties in the client-side registry; the YAML format may later declare properties for 74-series types without breaking the parser or the editor (consistent with FR-066).
- FR-020c: When the single selected component is an input switch built-in (FR-071c), the properties panel (FR-020a) shall present a control for its state offering the two values 1 and 0. Setting it changes the switch's persisted state, is an undoable action (FR-024), and is disabled while a simulation runs (FR-087); during a run the state is instead changed by clicking the switch (FR-087a). The switch's state is per-instance interactive state rather than a numeric property (FR-020b), so it is not an `overrides` entry. (Reworked 2026-06-17; supersedes the three-value 1/0/? selector.)
- FR-020d: When the single selected item is a wire (FR-027) or a bus (FR-035) rather than a component, the properties panel (FR-020a) shall present a read-only synthetic property sheet describing the conductor's two endpoints. Because the displayed designator label can change (FR-011b), each endpoint description shall be generated dynamically from current design state on every selection. A component-pin endpoint reads "`<designator> <pin name>`" (the pin's instance label, FR-011b, and the pin name). A bus-breakout tap (FR-043a) reads "`<group>[<bit>]`" using the bus's snapped pin-group name (FR-042) and the tapped bit index, falling back to the internal bus id when the bus is not group-snapped. A group-snapped bus endpoint (FR-042) reads "`<designator> <group name>`". A plain junction (FR-034) — which always ties two or more conductors and is therefore connected — reads "`junction (x, y)`" from its grid position. A genuinely dangling free end (FR-029) reads "`unconnected (x, y)`". A single selected **segment** (FR-031) shows this same sheet for its **parent conductor** (the segment belongs to one wire/bus). The sheet is informational only and offers no editable fields. When the selection is empty, holds more than one item, or is a single component, the prior behavior (FR-020a/FR-020c) applies. (Added 2026-06-25; reworked 2026-06-27 to read a plain junction as "junction (x, y)" rather than "unconnected (x, y)" — a junction is a connection of ≥2 conductors, so labeling it unconnected was wrong.)
- FR-105: When a component instance is selected, the properties panel shall present a read-only Documentation section sourced from the instance's type data (FR-104): the one-line description, a clickable link to the datasheet, and the per-pin roles. Each is shown only when present, and nothing is rendered for a type that carries no documentation.

### 3.7 The Canvas and Grid

- FR-021: The entire canvas shall be backed by a uniform fine grid (approximately 1–2 mm screen spacing at default zoom). All components, pins, wire endpoints, and bend points shall be located on grid intersections.
- FR-022: The user shall be able to zoom in and out on the canvas.
- FR-022a: The View menu (FR-004a) shall provide a **Fit to Screen** command that adjusts the viewport — zoom and pan together — so the entire design's geometry (all components, wires, and buses) fits within the canvas with a small margin, centered. Zoom is clamped to the allowed range (FR-022); when the design has no geometry the command is a no-op. It has no keyboard accelerator (FR-004b) and remains available while a simulation runs (FR-087), like the other zoom commands. The same fit is performed **automatically as the last step of loading any design** — File ▸ Open (FR-052) and hierarchy navigation into or back out of a sub-design (FR-100/FR-100a) — so a freshly loaded design arrives framed in the viewport. (Added 2026-06-28; auto-fit-on-load added 2026-07-06.)
- FR-023: The user shall be able to pan the canvas.
- FR-023a: In select-tool mode, a left press on bare canvas (a press that begins on empty canvas, not on a component, wire, bus, or pin) begins a rubber-band selection (FR-016b); a left press on bare canvas released without dragging clears the selection, or, with Shift held, preserves it. Left-dragging bare canvas no longer pans. Panning by middle-button drag and by Space+left-drag shall remain available in all tool modes (FR-023); the view may also be recentered by right-clicking bare canvas (FR-023b). (Reworked 2026-06-13; supersedes left-drag-on-empty-canvas panning, replaced by rubber-band selection in the KiCad style.)
- FR-023b: A right-click on bare canvas (no object under the cursor) shall recenter the view on the clicked point — the world point under the cursor becomes the new center of the canvas viewport, with the zoom level unchanged. Because a browser cannot warp the OS mouse cursor to the new center (as a native app such as KiCad does), the recenter shall instead arm a **pending zoom anchor** at the canvas center: until the pointer next moves, a wheel/trackpad zoom (FR-022) shall anchor at the canvas center rather than at the stale cursor position, so a zoom begun immediately after the recenter keeps the recentered point fixed. The pending anchor is cleared by the next pointer movement over the canvas. A right-click directly over an object instead opens that object's context menu (FR-033b). (Extended 2026-06-27 with the pending-anchor behavior; the prior recenter left zoom anchored on the off-center cursor, drifting the recentered point.)
- FR-024: Undo and redo shall be supported for all user actions that modify the design.

### 3.8 Wire Drawing

- FR-025: The application shall provide a Wire tool. While the Wire tool is active the cursor shall provide clear visual feedback indicating wire-drawing mode: a wire cursor drawn as a short diagonal line centered on the pointer, interrupted at its midpoint by a small open dot that marks the cursor's active point (the hotspot is the image center). The Wire tool's toolbar button (FR-026) shall display this same wire icon in place of a text label. (Reworked 2026-06-11; supersedes the uncentered lower-right→upper-left line, whose active point sat at an end of the glyph and read as a jump once a rubber band anchored to it.)
- FR-026: The user shall activate the Wire tool by clicking the Wire button in the top bar (FR-004a). (Reworked 2026-06-21; the former flat toolbar became the combined menu/tool bar, but the Wire tool remains a button, not a menu item.)
- FR-027: To draw a wire, the user shall click a source pin, then click a destination pin. The system shall create a wire between the two pins following the proposed route (FR-027c) current at the moment of the destination click. (Reworked 2026-06-12; supersedes the straight rat's-nest line — the committed wire now follows the Manhattan route proposal, with the straight line retained only as the router's fallback.)
- FR-027a: After the source is clicked and before the destination is clicked, the system shall display a rubber-band preview of the proposed route (FR-027c) from the source point to the current cursor position, recomputed as the cursor moves until the wire is committed or the gesture is cancelled. When intermediate waypoints have been locked (FR-027e), the already-locked legs (source→waypoint₁→…→last waypoint) are shown fixed, and only the live leg from the last waypoint to the cursor is recomputed as the cursor moves. The same preview applies to bus drawing (FR-039). (Reworked 2026-06-12; supersedes the straight-line-only preview, which survives only as the router's fallback per FR-027c. Extended 2026-06-19 for locked waypoints, FR-027e.)
- FR-027b: In select-tool mode, a component pin shall act as a wire hotspot, so the user can draw a wire without first activating the Wire tool. While in select-tool mode, hovering the cursor within a pin's hot region (FR-013d) shall change the cursor to the wire cursor (FR-025), and clicking there shall begin a wire from that pin exactly as the Wire tool does (FR-027/FR-027a); the wire is completed by the destination click and the application returns to select-tool mode (FR-028). Only pins are select-tool wire hotspots; clicks on wire segments, buses, bend points, and component bodies retain their existing select-tool meanings (FR-016, FR-031, FR-033b, FR-023a).
- FR-027c: The proposed route shall be a Manhattan (orthogonal) path from source to destination that avoids passing under component bodies and avoids lying on top of existing wires and buses (FR-027d), leaves a pin in the pin's facing direction (away from the component body), and prefers routes with few corners. Routing is best-effort: when no such route is found, the proposal shall degrade to the straight rat's-nest line. On commit, the route's corners shall become ordinary bend points, fully editable by the existing mechanisms (FR-031 through FR-033); a fallback straight proposal commits as a two-point wire, exactly as before.
- FR-027d: No wire or bus segment shall be routed coincident with (lying directly on top of) another wire or bus segment. Two conductors may *cross* — intersect transversely at a single grid point — but no length of one conductor may run collinearly over another. The proposed route (FR-027c) shall enforce this: an existing wire or bus segment occupies the unit grid *edges* it spans, and the router shall not traverse an already-occupied edge (sharing a single grid point, as at a crossing or a fan-out/junction vertex, remains permitted). Because this further constrains the orthogonal search, more wires will degrade to the straight rat's-nest fallback (FR-027c); the straight fallback is itself exempt from this rule. **Scope:** this constraint governs *automatic routing at draw time* only. It does **not** govern subsequent manual edits — dragging a bend point (FR-031/FR-032) or moving a component (FR-018) may still bring segments into coincidence, and the user is responsible for re-routing as today. Enforcing the no-overlap invariant on manual edits is a deferred follow-up (see design §6.9a) and is intentionally out of scope here.
- FR-027e: While a wire or bus is in progress (after the source, before completion), a single click on bare canvas — for the Wire tool, not on a pin or existing wire/bus segment; for the Bus tool, also not on a pin group or component body — shall lock an intermediate **waypoint** at the nearest grid intersection rather than completing the conductor. The proposed route (FR-027c) re-initializes from each locked waypoint (the waypoint imposes no escape direction), so the committed conductor is the concatenation of independently-routed legs source→waypoint₁→…→waypointₙ→destination, each leg degrading to its straight fallback independently. Any number of waypoints may be placed. On commit, the locked waypoints become ordinary bend points (FR-031–FR-033), indistinguishable from routed corners — they are not locked or otherwise special after completion. **Backspace** removes the most-recently locked waypoint (re-routing the live leg from the previous one); **Esc** cancels the entire in-progress conductor. A conductor can be **completed only on a real target** — a pin, an existing wire/bus segment, an existing dangling endpoint (FR-034c), or (for buses) a pin group or component body; it can no longer be terminated at a free point in empty space by clicking. (Added 2026-06-19. Supersedes the Bus tool's former behavior of ending a bus at a free dangling point on an empty-canvas click; a bus may still *start* at a free point.) (FR-039: buses follow the same model.)
- FR-028: After a wire is placed, the application shall automatically return to select-tool mode.
- FR-029: A wire or bus with exactly one connected endpoint shall be permitted (e.g., as a result of deleting a component).
- FR-030: A wire or bus with no connected endpoints shall be automatically removed from the design.

### 3.9 Wire Routing (Bend Points)

- FR-031: In select-tool mode, a plain click on a wire or bus segment shall select **that segment** — the single path edge between its two neighbouring bend/endpoint points — rather than the whole conductor, and the selected segment shall be highlighted on the canvas. Segment selection enables single-leg delete (FR-033d) and shows the parent conductor's property sheet (FR-020d). A press-and-drag beginning on a segment shall instead insert a bend point at the nearest grid intersection to the cursor, dividing the segment in two, and shall continue dragging the new bend until release (FR-032). Whole-conductor selection remains available via the rubber-band (FR-016b), and whole-conductor delete via the context menu (FR-033b). (Reworked 2026-06-28; supersedes selecting the whole conductor on click — segment-granular click matches KiCad and lets the user delete a single leg, FR-033d. Earlier rework 2026-06-10 superseded the original "click inserts a bend".)
- FR-032: The user shall be able to drag a bend point to any grid intersection while holding the mouse button down; the two segments touching the bend point shall rubber-band continuously during the drag.
- FR-032a: The user shall likewise be able to drag a junction point (FR-034b) — the dot where wires/buses branch — to any grid intersection while holding the mouse button down, exactly as a bend point (FR-032). Because a junction is a single shared electrical vertex (§7.1a), dragging it moves that vertex, so every conductor meeting at the junction follows and the branch stays connected; all segments touching the junction rubber-band continuously during the drag. The drag changes only geometry, never connectivity (the net is unchanged). (Added 2026-06-16; the prior implementation made junctions immovable because the bend hit-test ignored junction nodes.)
- FR-033: The user shall be able to right-click a bend point and select "Delete bend point"; the bend point shall be removed and the two segments it connected shall merge into one straight segment between the surrounding endpoints.
- FR-033a: In select-tool mode, the user shall be able to delete an entire wire or bus.
- FR-033b: A right-click directly on an object shall open a context menu offering the actions appropriate to the item under the cursor: a bend point offers "Delete bend point" (FR-033); a wire offers "Delete segment" (FR-033d, the segment under the cursor) and "Delete wire" (FR-033a); a bus offers "Delete segment" (FR-033d), "Set width…" (FR-038), "Edit bit names…" (FR-037b), and "Delete bus" (FR-033a); a component offers "Delete component" (FR-018a) — a sub-design instance additionally offers "Open sub-design" (FR-100), and a port carrying an off-sheet target additionally offers "Follow off-sheet connector" (FR-101). The menu is dismissed by choosing an item, pressing Escape, or clicking elsewhere. A right-click on bare canvas does not open a menu; it recenters the view (FR-023b). (Updated 2026-06-13: right-click on empty canvas now recenters instead of opening a menu.)

### 3.10 Wire Branching and Connectivity

- FR-033c: A bend point that does not actually bend the conductor shall not be created or retained by an editing action. **On completing a new wire or bus** (FR-027/FR-035 — including locked waypoints, FR-027e, and routed corners, FR-027c), the committed path shall be scanned and every interior bend point that is collinear with its two neighbours, and lies between them so its removal leaves the drawn shape unchanged, shall be dropped; this normalization is part of the single creation action (FR-024). **On concluding a drag of an existing bend point** (FR-032), if the bend's final position is collinear with its neighbours (as above), the bend shall be deleted (FR-033) rather than moved. **On concluding a drag begun on a straight segment** (FR-031) to insert a bend, if the released point is collinear with the segment's endpoints no bend shall be created. The rule applies identically to wires and buses (FR-039). It governs only these creation and drag actions; previously-saved conductors are not retroactively swept. Collinearity is tested with a small tolerance, and because all points lie on grid intersections (FR-021) it is exact in practice. (Added 2026-06-25.)
- FR-033d: In select-tool mode the user shall be able to **delete a single selected segment** (FR-031) of a wire or bus — via the Delete/Backspace key on the segment selection, or a "Delete segment" context-menu item (FR-033b) — distinct from deleting the whole conductor (FR-033a). Deleting segment *i* cuts the conductor's path between its two endpoints into the leading part (up to and including point *i*) and the trailing part (from point *i*+1 onward). Each side that retains at least two points survives as a conductor; a side reduced to a single point is dropped, leaving that lone endpoint (e.g. a component pin) simply unconnected. Where a cut falls on an interior **bend** point, that point is promoted to a **dangling free endpoint** (FR-029) at its coordinates; where it falls on an existing node (pin/junction/free) that node is retained, with junctions demoted and fully-disconnected conductors pruned by the ordinary cleanup (design.md §3.3 G2 / FR-030). Thus deleting an **interior** segment yields two conductors each with one dangling end, while deleting a **first/last** segment yields one shorter conductor with a dangling end plus a now-unconnected endpoint. The whole deletion is one undoable action (FR-024). For a bus the two parts inherit the bus's width and bit names, and each retained `groupConnection` (FR-042) stays with whichever part keeps its endpoint vertex. Rule applies identically to wires and buses (FR-039). (Added 2026-06-28.)
- FR-034: While the Wire tool is active, clicking on an existing wire segment shall start a new wire branch from that point rather than inserting a bend point.
- FR-034a: A single pin shall be permitted to have more than one wire connected to it (fan-out), so that one output may drive multiple inputs.
- FR-034b: A point at which a wire branches from another wire (FR-034) constitutes an electrical junction. All pins and wires that are transitively connected through pins and junctions form a single electrical net. The design's connectivity (its set of nets) shall be derivable from the saved design without reference to pixel geometry.
- FR-034c: Beginning or completing a new wire or bus **on an existing dangling endpoint** (a free, unconnected end, FR-029) of the **same conductor type** — and, for a bus, of the **same width** (FR-039a) — shall **join** the two conductors into one continuous conductor rather than creating a junction (FR-034b) or leaving any connection mark. The two coincident endpoints merge into a single interior point of the joined conductor; that point becomes an ordinary bend point and is then subject to collinear-bend normalization (FR-033c), so a straight-through join leaves no bend at all. No junction vertex and no connection dot result, and the dangling-end mark (FR-029) disappears. The join applies only when the target end matches: a wire completed on a dangling **bus** end still taps a bit (FR-043b), and a bus completed on a **width-mismatched** bus end is rejected (FR-039a). (Added 2026-06-28.)

### 3.11 Bus Drawing

- FR-035: The application shall provide a Bus tool, separate from the Wire tool.
- FR-036: Buses shall be rendered as thick blue lines. Single-bit wires shall be rendered as thin black lines.
- FR-037: Each bus shall display a width annotation consisting of a slash mark and a digit indicating the bus width in bits.
- FR-037a: A bus of width N shall represent N independent single-bit signals (nets). Signal identity is determined by bit position: bit i of a bus is a distinct net from bit j (i ≠ j). The connectivity rule of FR-034b applies independently to each bit.
- FR-037b: A bus may optionally carry a signal name for each bit (e.g., C, V, N, Z). When a bus is snap-connected (FR-041) to a pin group whose member pins are named, and the bus does not already have bit names, the bus shall adopt the group's pin names in bit order. Bit position, not name, determines electrical connectivity.
- FR-038: The user shall be able to right-click a bus and set its width from a context menu.
- FR-039: Bus drawing, bend-point editing, and branching shall follow the same interaction model as wires (FR-026 through FR-034).
- FR-039a: A connection that would join two buses of unequal width shall be prevented at the time the user attempts it.
- FR-040: After a bus is placed, the application shall automatically return to select-tool mode.

### 3.12 Bus-to-Component Snap Connection

- FR-041: When the user drags a bus endpoint onto a component, the system shall determine which of the component's declared pin groups can **accept** the bus. Each pin carries one bit. A pin group accepts a width-`w` bus when, scanning its member pins in declared bit order and excluding any pins already claimed by an existing group connection on that component+group, there is a **contiguous run of at least `w` currently-unconnected pins**; the bus then claims the first `w` pins of the lowest such run (pack-low, FR-041c). (Reworked 2026-06-20: previously a group "matched" only when its member-pin count *equaled* the bus width; a narrower bus may now connect to a free sub-block of a wider group.)
- FR-041a: If exactly one pin group accepts the bus, the system shall snap-connect the bus to that group automatically (per FR-042/FR-041c).
- FR-041b: If more than one pin group accepts the bus, the system shall prompt the user to choose which group to connect to, presenting the candidate groups by name; the user may cancel the connection. Within a single group the placement is fixed by the pack-low rule (FR-041c), so the prompt distinguishes groups, not positions within a group. (This supersedes any rule that silently selects a default group on a tie.)
- FR-041c: A bus narrower than a pin group may connect to a contiguous block of currently-unconnected pins within that group, provided the block is at least the bus width. Placement is **pack-low**: the bus claims the first (lowest, in declared bit order) free block of sufficient size, taking exactly its own width's worth of pins from the low end of that block; bit `i` of the bus connects to the `i`-th pin of the claimed block (FR-042). This allows, e.g., two 4-bit buses to share one 8-pin group (the first claims pins 1–4, the second pins 5–8), and disallows connecting an 8-bit bus once any pin of an 8-pin group is taken (no 8-pin free block remains). A group with no free block of at least the bus width does not accept the bus (FR-043). Equal-width connection is the trivial case (the whole group is one free block). (Added 2026-06-20.)
- FR-042: When a pin group is connected — automatically per FR-041a or by the user's choice per FR-041b — the system shall connect each bit of the bus to the corresponding pin of the claimed block (FR-041c) in the group's declared bit order, without requiring the user to wire each pin individually (the claimed block is the whole group for an equal-width connection). The snapped bus endpoint is electrically connected even though its endpoint vertex remains `free` (the binding is recorded as group metadata, not geometry); it follows its component when that component is moved or rotated (FR-018/FR-018c). A group-snap connection shall be shown with a large **curly-brace** glyph rather than a per-vertex mark (FR-042a).
- FR-042a: A group-snap bus connection (FR-042) shall be rendered as a large curly brace enclosing the connected pins — the **claimed block** (FR-041c), which is a sub-range of the group for a partial connection and the whole group for an equal-width one: the brace opens toward the pins, its two tips touching the visual attachment points of the claimed block's two outermost pins. The brace is formed of two halves — an upper and a lower spline — that meet just outside the block's center, away from the component body, **coming to a point** there (the apex); the bus shall visually terminate at that apex point, and on connecting, the bus endpoint shall be positioned at the apex (a geometry placement only — the connection itself is the group binding, FR-042). The apex shall lie on a **grid point**: its position along the pin row is taken from the claimed block's middle pin (`floor(k/2)`, `k` = block size) rather than the exact tip midpoint, and its outward offset is an integer, so it falls on a grid intersection even when the block has an even pin count — at the cost of the two brace halves being slightly asymmetric. The brace is recomputed from the connection's claimed pins each frame, so its orientation follows the group's pin side and the component's rotation and it tracks the component on move/rotate. A genuinely unconnected (dangling) free endpoint keeps the FR-029 dangling mark; every connected bus is always drawn with its brace.
- FR-042b: While the bus tool is active, the proposed group-snap brace (FR-042a) shall be shown as live feedback whenever the mouse pointer comes near a pin group that can **accept** the bus (FR-041/FR-041c) — within a small range, **without requiring a click within the component body** — choosing the group nearest the pointer when several are in range (FR-041b). The brace previews the exact pins the bus would claim (the pack-low block, FR-041c), so a partially-filled group shows a brace over its next free block; a group with no free block of sufficient size shows none. Before the first click the brace alone is shown; while a bus is in progress the in-progress bus is also drawn terminating at the prospective apex. When the in-progress bus's **source** endpoint is itself a group snap, that source group's brace shall remain shown for the duration of the drag (not just while the pointer is near it), so the originating connection stays visible while the route is dragged out; both the source brace and any prospective destination brace are drawn together. A click anywhere the brace feedback is showing shall begin (first click) or terminate (second click) the bus at that group's apex; it shall not be necessary to click on the part body to start or end a group-connected bus. (FR-042a/FR-042b added 2026-06-18, replacing the interim blue-diamond mark; apart from placing the endpoint at the apex this is a rendering/feedback feature and does not change the connectivity representation.)
- FR-042c: Before the bus's **first** endpoint is placed the bus has no committed width, so the group-proximity feedback and first-click snap (FR-042b) shall match a pin group of **any** width (the group nearest the pointer), and starting the bus on such a group shall set the bus width to that group's available free-block size (FR-041c) — the whole member-pin count for an untouched group. Once the first endpoint fixes the bus width, group proximity reverts to the strict width-matching of FR-041/FR-042b for the second endpoint. This lets a fresh bus adopt the width of the group it is started from, instead of matching only groups of the default width. (Added 2026-06-19; previously the first endpoint also matched only default-width groups, so a bus started near, e.g., a 4-wide group showed no feedback.)
- FR-043: If no pin group accepts the bus (FR-041/FR-041c — none has a free contiguous block of at least the bus width), the bus endpoint shall be left unconnected. (Supersedes the earlier rule that attached the endpoint to the nearest pin only: an unmatched drop now connects nothing rather than guessing a single pin.)
- FR-043a: The user shall be able to break out a single signal (one bit) from a bus and route it as an ordinary single-bit wire to a pin or other connection point. The broken-out wire shall be electrically part of that bus bit's net (per FR-037a).
- FR-043b: Symmetrically, the user shall be able to draw an ordinary wire from a pin (or other connection point) and **terminate it on a bus**: clicking a bus segment as the in-progress wire's destination shall present the **same** bit-choice dialog as starting a breakout from the bus (FR-043a), and on confirmation create the single-bit breakout connection (FR-043a) at the clicked point. The resulting wire shall follow the route already drawn (its locked waypoints and routed corners, FR-027e) — unlike a breakout *started* on a bus, which commits straight (FR-043a) because no route has yet been drawn — so the commit matches the live preview. Cancelling the dialog shall leave the wire still in progress (no connection made, the gesture not ended). (Added 2026-06-23.)

### 3.16 Design Save Format

- FR-055: Designs shall be saved as JSON files.
- FR-056: The JSON file shall contain at minimum three distinct collections: (a) component instances, (b) wire routes, and (c) bus routes.
- FR-057: Each component instance record shall include: its reference designator (`refdes`, the immutable internal identity, FR-011), the component type's identity (its type id, FR-066e), an optional display label (FR-011b; absent means it defaults to the `refdes`), canvas position, rotation, and a full copy of the type's data (captured from the library at placement time and persisted verbatim at save; only Refresh Types, FR-088, re-copies it — clarified 2026-07-08, superseding "from the YAML file at the time of save", which wrongly implied a re-copy at save). (Sub-design instances are an exception per FR-098: they store a relative child-file path and a chosen render style in place of any copied type data.)
- FR-058: Per-instance overrides of type data (e.g., a custom propagation delay for a specific instance) shall be stored alongside the copied type data in the instance record.
- FR-059: Each wire route record shall include: the two endpoint references and an ordered list of bend-point grid coordinates. An endpoint reference shall be one of: (a) a component pin (U-number and pin name), (b) a junction on another wire or bus (FR-034b), or (c) a free canvas grid coordinate if the endpoint is dangling.
- FR-059a: The saved design shall represent electrical connectivity (the set of nets per FR-034b) in a form derivable without reference to pixel geometry, so that a later tool can determine which pins are electrically connected.
- FR-060: Each bus route record shall include: the same endpoint and bend-point data as a wire, plus the bus width in bits and any pin-group connection data for snap-connected endpoints.
- FR-060a: For buses, the saved design shall additionally include any per-bit signal names (FR-037b) and any single-bit breakout connections (FR-043a), such that the net membership of each individual bus bit — including which bus and bit each net originates from — is derivable without reference to pixel geometry (consistent with FR-059a).
- FR-060b: The save format shall additionally represent: (a) port objects — label, effective direction (FR-094c/FR-094d), an optional direction override (FR-094d), and optional off-sheet target `{file, label}` (FR-094/FR-101) — as instances whose connection points are `connector` vertices (design.md §7.1a), a multi-bit portN (FR-071e) instead persisting its placement-fixed bit width and joining nets through its ordinary pins; (b) sub-design instances — X-series refdes, relative child path, chosen render style, position, rotation, and the wiring to their interface pins (FR-098/FR-099) — with **no** copied child type data (superseding FR-057 for them); and (c) the design-level default embed rendering (FR-096). Connectivity (FR-059a) shall remain derivable without pixel geometry, with connector vertices contributing to nets by label (FR-094a) and explicit cross-file links (FR-101a).
- FR-060c: The save format shall carry a numeric `formatVersion` identifying the format the file was written in. On load the editor shall provide compatibility across format versions: a file from an *older* version shall be migrated forward to the current version before use, by applying an ordered chain of single-version upgrade steps; if a required upgrade step is unavailable the load shall be rejected with a legible error rather than misread. A file from a *newer* version than the editor understands shall be loaded best-effort with a warning (forward-compatibility). A file with no `formatVersion` shall be treated as the oldest version. This is the compatibility path for evolving the save format as the product grows; the current version number and its migration chain are recorded in design.md §7.4.

### 3.17 Component Definition (YAML File)

- FR-061: Each TTL component type shall be defined by a YAML file (`.yaml` extension), parsed by the server. (The format was designed collaboratively; see design.md §7.6.)
- FR-062: The YAML file shall specify: component type name; the rectangular outline dimensions (either stated directly or derived from the author-placed pins per FR-062b); and for each pin: its name, the side of the rectangle it appears on (left, right, top, bottom), and its position along that side. Power and ground pins shall not be represented in the YAML file, the editor, or the simulation.
- FR-062a: The YAML file shall specify each pin's electrical direction, which shall be one of exactly: input, output, bidirectional, or tristate-capable, so that high-impedance behavior and signal direction can be represented in a later simulation phase. There is no power/ground direction (see FR-062).
- FR-062b: Outline dimensions may be stated explicitly in the YAML file; if omitted, the server shall derive the outline from the author-placed pins. Each pin may optionally carry a physical pin number, which is footprint/BOM metadata only and shall not be used for drawing or simulation. (Supersedes the earlier declared-package mechanism: the standard-physical-package keyword, the package-name grammar, and the package table/generator are removed entirely, because power/ground — the only reason the physical package affected the symbol — are no longer represented.)
- FR-062c: The YAML file shall optionally specify `rendertype` (`unit` — the default — or `subunit`). For `rendertype: subunit` it shall additionally specify `numunits` (the number of functional units in the package) and `renderas` (the schematic symbol, per FR-013b), and each pin shall specify its `unit` (FR-014a) in place of a position. A `unit`-rendered component is drawn as the rectangle of FR-013.
- FR-062d: The YAML file shall optionally specify `clock: <pin name>`, naming the input pin that clocks the component's registered (`.R`) behavior outputs (FR-079). The parser shall verify the named pin exists and has direction input. The single `clock:` pin is the global-clock model of real GALs (the GAL16V8/GAL20V8/GAL22V10 devices, FR-066a); it is required for any `.R` output that does not name its own clock. In the extended dialect (FR-079a) a registered output may instead carry a GAL20RA10-style `.CLK` equation naming its own clock, so a part with independent clock domains needs no global `clock:` for those outputs. (This makes explicit the named-clock convention first established in 74574.yaml's behavior comment.)
- FR-062e: The YAML file may optionally include a top-level `physical:` block recording physical-package metadata for use **only by netlist exporters** (KiCad, NDL, BOM tools): it shall be carried verbatim onto the `ComponentType` (and thus copied into saved designs per FR-057) and shall be ignored by the editor and both simulators — power and ground remain unrepresented there (FR-062 is unchanged). The block contains: `pincount` (required — the package's total physical pin count), `package` (optional free-form package name, e.g. `"DIP-14"`; no naming grammar is imposed and the server never interprets it, so this does not revive the package mechanism removed by FR-062b — the block can never affect the symbol, the editor, or the simulation), `power` (a list of `{name, number}` entries naming the power/ground pins; the `name` — e.g. `VCC`, `GND` — is the rail net label an exporter attaches the pin to, and several entries may share a name), and `nc` (optional list of physically no-connect pin numbers). When the block is present the server shall validate **physical completeness**: every signal pin in `pins[]` carries a `number`; the signal, power, and nc numbers are mutually distinct; together they account for exactly the numbers 1..`pincount`; and no `power[].name` collides with a `pins[].name`. A violation rejects the component (skipped with a logged error at load, consistent with FR-063a). When the block is absent nothing changes: `pins[].number` remains optional footprint/BOM metadata (FR-062b) and exporters degrade gracefully (warn, or invent numbering where their format demands one). (Added 2026-07-06.)
- FR-063: The YAML file shall optionally specify one or more named pin groups, each listing the pins that form a bus and their bit order, to support bus snap-connection (FR-041 through FR-043).
- FR-063a: A pin group (FR-063) shall be **geometrically coherent** so the bus snap UI — in particular the curly-brace indicator (FR-042a), which assumes the group's pins are colinear along one edge — renders and targets correctly. Two constraints hold: (1) **same side** — all member pins lie on one side of the component; and (2) **contiguous** — no pin that is *not* a member lies between the group's members on that side (members occupy a consecutive run by `pos`, for `unit` components). Empty positions between members are allowed; only an intervening non-member pin is not. The server shall **reject** a component whose any group violates this (skipped with a logged error at load, consistent with other parse failures), and the New GAL part dialog (FR-066d) shall **prevent** defining such a group. This constrains only the symbol layout (`pos`/`side`): physical pin numbering (`number`) is independent, so a part whose silicon interleaves two buses is drawn with each bus contiguous on the symbol while `number` preserves the true pinout.
- FR-064: The YAML file shall optionally specify propagation delay values for the component.
- FR-065: The server shall expose the parsed component library to the browser application via an API endpoint.
- FR-066: The YAML file format shall accommodate later addition of behavioral logic equations (in GALasm form, per the vision statement) to a component definition without requiring changes to the editor or breaking the existing parser. The editor phase shall ignore any behavioral content present.
- FR-066a: A component definition may optionally declare `gal: <device>`, naming the GAL device whose GALasm dialect and physical capacity the behavior block (FR-079) must conform to. The value is one of GALasm's own device names — `GAL16V8`, `GAL20V8`, `GAL22V10`, or `GAL20RA10` (galasmManual.txt). Declaring it selects **strict** simulation of the behavior block (FR-079b): the slow simulator accepts only that device's language subset and per-device capacity and rejects anything outside it. Omitting the key — the default for the 74-series library, whose parts are not GALs — selects the **extended** dialect (FR-079a), under which the behavior is modeled freely with no device-capacity limit. The server validates only that a present `gal:` value is one of the four names; the dialect it selects is enforced client-side by the simulator (FR-066 keeps the behavior block otherwise opaque to the server). The key is additive and ignored by the editor phase (consistent with FR-066).
- FR-066b: A GAL part (a type declaring `gal:`, FR-066a) shall additionally declare a `partnumber`: a non-empty string naming that specific programmed part (e.g. "PC-DECODE-A"), distinct from `type`, which for a GAL names only the device family (e.g. "22V10"). The part number is the GAL part's free-form **display name** (FR-005b): it is shown as the chip's type label on the canvas and as the palette tile's button text, and the user may edit it freely. It is **not** the library key — that role belongs to the type's immutable `id` (FR-066e), so two parts of the same device family coexist by virtue of distinct ids, and a part number need not be unique. A GAL part may also carry a per-part `description` (the same one-line field as FR-104), since a programmable part has no datasheet description of its own; it is authored with the part (FR-066c) and surfaced per FR-005b/FR-105. A placed GAL instance records its type's `id` (FR-066e) as its instance type identity and shows its `partnumber` as the chip's type label. Non-GAL (74-series) types are unaffected: they have no `partnumber` and their display name is the `type` (FR-005). (Reworked 2026-06-22; previously `partnumber` was the library key and YAML filename stem and was required library-unique — those roles moved to the new immutable `id`, FR-066e, leaving `partnumber` a free-form label.)
- FR-066c: The application shall provide a **"New GAL part"** authoring dialog that produces a valid GAL component definition without hand-editing YAML. The dialog presents the device's fixed physical skeleton (initially the 24-pin GAL22V10: pin 1 clock/input, pins 2–11 and 13 inputs, pins 14–23 the OLMC I/O pins, pin 12 GND, pin 24 VCC) and collects only what varies between parts: the `partnumber` (FR-066b), an optional one-line `description`, a label for each I/O-capable pin, a per-OLMC direction (input / combinational output / registered output), optional named pin groups (FR-066d), and the GALasm `behavior` block (FR-079). The behavior field is validated **live** against the device using the existing strict validator (FR-079b, `gal: GAL22V10`) — the same accept/reject gate Run applies — so the dialog cannot emit a part that violates the device's language subset or capacity. Accepting the dialog creates the part (FR-007a); cancelling discards it. Creation is one-shot: the application returns to select-tool mode (consistent with FR-010). The dialog is reached from the upper palette region (placement of authored parts is then ordinary FR-008/FR-009 placement).
- FR-066d: The New GAL part dialog (FR-066c) shall let the user define the part's named pin groups (FR-063) — the bus interfaces a multi-bit bus can snap-connect to — through a **"Pin groups…"** button that opens a modal sub-dialog. The sub-dialog shall list the part's already-defined groups, each removable, and shall let the user define a group by entering a name and selecting a subset of the part's pins (by checkbox, shown by their current labels). A part may have multiple groups, added across repeated openings of the sub-dialog. A group's member order — the bus bit order (FR-063) — shall follow the pins' physical layout order on the part (top-to-bottom). Group membership shall track the underlying skeleton pin, not the label text, so renaming a pin after grouping it does not invalidate the group; the emitted YAML resolves members to their current labels. A group name shall be non-empty and unique within the part, a group shall have at least one member, and the selected pins shall satisfy the same-side and contiguity constraints of FR-063a (the dialog rejects a selection that would straddle two sides or be interrupted by a non-member pin). Pin groups do not affect the strict behavior validation (FR-079b).
- FR-066e: Every component type shall carry an immutable, library-unique internal identifier `id` (e.g. "type-74125", "type-22V574"), divorced from its human-readable display name (FR-005/FR-005b). The `id` is the sole key by which the library, the palette, placed instances (their `type` field, FR-057), the simulator's per-type behavior cache, and Refresh Types (FR-088) refer to a type. The display name — the `type` for a 74-series part, the `partnumber` for a GAL part — is by contrast completely free-form and may be changed by editing the YAML without affecting the `id` or breaking any placed instance. The TTL library's `id` values are authored explicitly in each file; a GAL part created in-app (FR-066c/FR-007a) is assigned its `id` at creation, from which the YAML filename is derived. The `id` key may be **omitted** from a YAML file, in which case the parser derives it as `type-` + (`partnumber` ‖ `type`) — keeping the format additive (FR-066) so pre-existing or hand-written files still load; an author who wants an `id` that stays fixed across display-name edits states it explicitly (as every library file does). The server shall key its in-memory library by `id` and shall reject a create (FR-007a) whose `id` collides with an existing part. (Added 2026-06-22; this supersedes the part of FR-066b that made `partnumber` the GAL library key and filename stem, and the part of FR-005 that keyed 74-series parts by `type`.)
- FR-104: The YAML file may optionally carry human-readable documentation, all fields optional: a one-line `description` (the part's function), a `datasheet` block (manufacturer/`vendor`, document `title`, `rev`, and `url`), and a per-pin `desc` (the pin's role). Documentation is presentation-only; it never affects outline, pins, geometry, or simulation. Absent documentation is normal (FR-066).

### 3.17a Memory Device Generator (RAM/ROM)

- FR-114: The application shall provide a **memory-device generator**, reached from a **"NEW MEM"** action button in the **upper** palette region (alongside the New GAL part action's **"NEW GAL"** button, FR-066c): clicking it raises a **New memory device** dialog that specifies RAM and ROM components without hand-editing YAML. A memory device so defined is a placeable component that appears in the **upper** palette region (like an authored GAL part, FR-007a/FR-066c) yet whose simulation semantics are a **built-in behavior** supplied by the client application (the built-in analogue of FR-067a/§3.19), not GALasm equations — it is a generated metatype, like the GAL generator, but backed by a built-in behavior rather than by a GALasm behavior block. The dialog's top control is a **RAM / ROM** radio (default **RAM**); changing the selection **completely re-initializes** the dialog to the chosen class's control set. The dialog collects a free-form **name** for the device (FR-114a/FR-114c) and configures the device **size** — the number of addressable locations, a power of two, entered as a count of **address bits** *n* (the device has 2ⁿ locations) — and the **data width**, one of 4, 8, 16, or 32 bits. A **ROM** additionally specifies a **content file** defining its stored data; a **RAM** has no content file. Accepting the dialog creates the device; cancelling discards it. Creation is one-shot, returning to select-tool mode (consistent with FR-066c). (Added 2026-06-23.)
- FR-114a: In the New memory device dialog (FR-114): a free-form **name** shall be required (non-empty) and becomes the device type's display name (FR-114c); the name field is pre-filled with a size-based suggestion (e.g. "RAM 256×8") that keeps tracking the class/size/width until the user edits it, after which the user's text is left untouched. The **address-bit count** *n* shall be a positive integer in the supported range (1 ≤ *n* ≤ 24, i.e. up to 16,777,216 locations) and shall be shown alongside the resulting location count (2ⁿ) for confirmation; the **data width** shall be chosen from exactly {4, 8, 16, 32} bits (default 8); and for a **ROM** a **content file** must be selected before the device can be created, while a **RAM** offers no file control. The ROM content file shall be chosen through the **server-side file-navigation dialog** (FR-053, the same browser used by Open). Switching the **RAM/ROM** radio re-initializes the class-specific controls to their defaults — switching to RAM removes the file control and discards any chosen file; switching to ROM presents the file selector. The dialog shall disable its **Create** action until the current class's inputs are valid.
- FR-114b: The **pinout** of generated memory devices is specified in FR-114c, their **built-in simulation behavior** in FR-114d, the **ROM content-file format and loading** in FR-114e, and their **cross-session persistence** in FR-114f (all implemented). A generated memory metatype is now persisted and registered exactly like an authored GAL part (FR-007a): on creation the client writes it to the component library as a `.yaml` artifact carrying a `mem` block (FR-114f), so it survives reload and Refresh Types (FR-088). (A placed instance also still round-trips through a design save via its embedded type data, FR-057.) (Reworked 2026-06-25; previously this was the one deferred aspect, tracked by OQ-013, with generated types held in-session only. See design.md §6.4/§6.11/§7.6.)
- FR-114c: A generated memory device (FR-114) is drawn as an IC-style rectangle whose pins are fixed by the device class and the dialog's parameters (address bits *n*, data width *w*). **Common to RAM and ROM:** *n* address inputs `A0`…`A(n-1)` forming one snap-connectable pin group **ADDR** (FR-063); *w* data pins `D0`…`D(w-1)` forming one snap-connectable group **DATA**; an active-low chip-enable input `CE/`; and an active-low output-enable input `OE/`. **RAM** additionally has an active-low write-enable input `WE/`. The data pins are **bidirectional** on a RAM (driven on read, sensed on write) and **tristate** outputs on a ROM (FR-062a). **Layout:** the ADDR group occupies the top of the **left** edge in order A0→A(n-1), with `CE/`, `OE/`, and — on a RAM — `WE/` placed below it on that same left edge (the control pins follow the address run and so do not interrupt it, FR-063a); the DATA group occupies the **right** edge in order D0→D(w-1). The device has **no power or ground pins** (FR-062). Active-low pins are labeled with a trailing slash (`CE/`, `OE/`, `WE/`). The device's free-form **name** (FR-114a) is its display name and derives its immutable library id `type-<name>` (FR-066e), so two devices with different names are distinct types regardless of identical pinouts; creating a device whose name (hence id) **collides** with an existing type — a loaded part, a GAL part, a built-in, or an earlier memory device — shall be **rejected** with an inline error, leaving the dialog open. The collision is detected by the server's create endpoint (FR-007a/FR-114f) — a duplicate `id`, or a library file of that name already present — returned as a conflict and surfaced inline; built-in ids, which are not in the server library, are additionally guarded client-side before the submit. (Added 2026-06-23; reworked 2026-06-25 — the rejection is now the persisted server check per FR-114f, superseding the in-session-only analogue.)
- FR-114d: The built-in simulation behavior of a generated memory device (FR-114c) is: **Deselected** — while `CE/`=1 the device neither reads nor writes and all data pins are high-impedance (Z). **Read** — while `CE/`=0 and `OE/`=0 (and, on a RAM, `WE/`=1) the data pins drive the stored word at the address present on `A0`…`A(n-1)`; otherwise the data pins are Z. **Write (RAM only)** — a RAM latches the value on its data pins into the addressed location on the **rising edge of `WE/`** (its 0→1 transition); the address shall be held stable while `WE/`=0, and while `WE/`=0 the data pins are disabled (Z) regardless of `OE/`, so the chip does not drive the bus from which it is being written. A **ROM** ignores writes (it has no `WE/`) and only ever reads. A read of a location whose contents are undefined (a RAM location not yet written) yields U on those data bits, consistent with the four-state value model (FR-077). Timing follows the unit-delay model (FR-078). The settled realization: a RAM **powers up with every cell undefined** (all reads U until written); reads and output-enable are evaluated against the **previous** step's net values and a write **latches** the address and data present at the WE/ rising edge (so the device responds one unit after its inputs); a high-impedance (Z) input is read as U (FR-077); any deciding control that is U (an uncertain `CE/`/`OE/`/`WE/`, or an `OE/`/`CE/` floating to Z) makes the device drive **pessimistic U** on the data bus rather than Z; and an address with any U/Z bit is undecodable and reads U (and drops a write). A **ROM** drives its loaded content (FR-114e); an address beyond the loaded content, or a ROM whose content file failed to load, reads U. (Added 2026-06-23.)
- FR-114e: A ROM's content file (FR-114a) defines its stored data as a sequence of **bytes in memory order**, in one of two formats selected by the file's **extension** (required): **`.bin`** is raw binary bytes; **`.hex`** is ASCII hex — whitespace-separated byte tokens (each one or two hex digits, value 0–255), where any run of spaces, tabs, CR, or LF separates tokens (consecutive whitespace collapses to one separator). Any other extension is rejected. Bytes pack into the device's `dataWidth`-bit locations **little-endian** — `ceil(w/8)` bytes per location, low byte first, masked to `w` bits (so a width-4 device uses the low nibble of its one byte; width 8 = one byte; width 16/32 = 2/4 bytes LE) — with file byte 0 forming location 0. A trailing partial word is dropped, and content beyond the device's capacity is ignored (and reported, FR-074). The content is read from the file **at Run** — so editing the file and re-running reloads it, and only the file **path** (not the data) is saved with the design (FR-060) — by the server (a localhost file read, since the browser cannot read an arbitrary path); a content file that is missing, the wrong type, or malformed is reported via the message tray (FR-074) and that ROM reads U, rather than aborting the run. (Added 2026-06-23.)
- FR-114f: A generated memory device (FR-114c) shall be **persisted to the component library** on creation, through the same create endpoint and rules as an authored GAL part (FR-007a). The client serializes the generated type to a component-definition YAML carrying its explicit pinout (pins and the ADDR/DATA groups), its `id` (`type-<name>`, FR-066e) and display name, and a **`mem` block** recording the class (`kind: ram|rom`), the address-bit count, the data width, the location count, and — for a ROM — the content-file path (FR-114e). The server parses this YAML like any component, carries the `mem` block onto the type, and serves it from `/api/v1/components`, so on reload the device reappears in the palette (FR-006a), places, and simulates from its `mem` block (FR-114d) with no in-app re-creation — the built-in behavior binds from the serializable `mem` data, not from session-only code. Duplicate-`id` and existing-file collisions are rejected as for any create (FR-007a), surfaced inline in the New memory device dialog (FR-114c). The ROM content file itself is **not** copied into the library; only its path is recorded, and the bytes are still read at Run (FR-114e). (Added 2026-06-25.)

### 3.19 Slow (Debug) Simulator

- FR-075: The application shall provide a slow ("debug") simulator, implemented in JavaScript in the browser front end, that executes the current design live on the editing canvas. It interprets the behavior of every component participating in the schematic: GALasm behavior blocks for 74-series types (FR-079) and registered behaviors for built-ins (FR-067a). The fast engine (C-code generator) is a future phase; nothing in the slow simulator's design shall preclude it.
- FR-076: The top bar (FR-004a) shall provide a button labeled "Run". Clicking it starts the simulation, relabels the button "Stop", and sets the state tray (FR-073) to "simulating". Clicking "Stop" ends the simulation, restores the "Run" label, and returns the state tray to "editing"; a run never terminates on its own (FR-085's oscillation bound pauses evaluation without ending the run — the stale "automatic termination" clause was removed 2026-07-08, a leftover of the auto-terminating combinational model FR-085 superseded). Starting a run clears the message tray (FR-074) of any stale editing-time message before the run's own start-up reports (FR-080, bus conflicts) are posted; stopping a run clears the message tray again (e.g. of the selection-lock notice, FR-087).
- FR-077: The simulator shall represent exactly four signal values on every net: logic 0, logic 1, U (undefined), and Z (high impedance, a tristate net with no enabled driver). A component reading Z from a net treats it as U. Logical combination shall be selectively pessimistic, as real logic permits: an AND with any 0 operand yields 0 and an OR with any 1 operand yields 1, regardless of U operands; every other combination involving a U operand yields U. (Reworked 2026-06-12; supersedes strict pessimism — *any* U operand → U — under which registered feedback could never be initialized: with `0 AND U = U`, even a held synchronous clear or load left a counter's registers U forever, making sequential design impossible.)
- FR-078: The simulator shall use a unit-delay timing model with 1 unit = 1 simulated nanosecond. At each unit step, every component computes its outputs from the net values of the previous step, and all nets then update simultaneously (double-buffered evaluation). Every component's outputs thus respond exactly one unit after its inputs, regardless of internal complexity, so logic settles in a deterministic, observable sequence; the YAML propagation delays (FR-064) are not used by the slow simulator.
- FR-079: For 74-series types, the simulator shall parse and evaluate the YAML behavior block per the GALasm reference (`galasmManual.txt`): flat sum-of-products equations; the declaration/use polarity XOR rule; plain, `.T` (tristate), and `.R` (registered) outputs; single-term `.E` enables; `AR`/`SP`; and the `VCC`/`GND` constants. `.R` outputs latch their sum-of-products value on the rising edge (0→1 transition between consecutive steps) of the component's declared clock pin (FR-062d). All registers power up as U.
- FR-079a: **Extended GALasm dialect (default).** When a 74-series type does not declare `gal:` (FR-066a), its behavior block is evaluated in the *extended* dialect: the union of all four GAL device dialects (galasmManual.txt) with their physical-capacity limits (pin/OLMC counts and per-OLMC product-term profiles) not applied — consistent with the long-standing stance that a behavior block adopts the GALasm *language*, not a GAL22V10's capacity (manual §5) — **plus** one genuine superset extension, an infix XOR operator. Beyond flat sum-of-products (FR-079), the extended dialect therefore provides: (a) an **XOR** operator — the one construct no real GAL equation language has (the 22V10's XOR fuse sets output *polarity* only, already modeled by the LHS-negation rule of FR-079), so XOR exists only in extended mode and never in strict (FR-079b) — letting parity functions such as the 74HC283 adder's sum bits be written directly instead of expanded to sum-of-products; and (b) the GAL20RA10 per-output **`.CLK`/`.ARST`/`.APRST`** suffix equations, giving each registered output its own clock and asynchronous reset/preset, and thus **independent clock domains** within one part (e.g. the dual-clock 74HC74, 74HC192, 74HC193, and 74HC595). A registered output that names no `.CLK` continues to use the part's global `clock:` pin (FR-062d). Behaviors that no GAL device can express — notably asynchronous parallel load of *variable* data, as on the 74HC165 (`AR`/`SP`/`.ARST`/`.APRST` only force a fixed 0 or 1) — remain outside even the extended dialect and are not addressed by this mechanism. (The 74HC165's *buried shift stages* — the seven internal flip-flops it does not expose — are separately addressed by FR-079c; only its variable async **load** stays outside the dialect, so the library part models parallel load *synchronously*, on a clock edge, per FR-079c.)
- FR-079b: **Strict device validation.** When a type declares `gal: <device>` (FR-066a), the slow simulator shall validate its behavior block against exactly that device's GALasm language subset and physical capacity, and refuse to run — a Run-preflight failure reported via the message tray (FR-074) alongside the other preflight checks of FR-079 — on any violation: a construct the named device lacks (e.g. the XOR operator on any device; `AR`/`SP` on a GAL16V8/GAL20V8/GAL20RA10; `.E` on a GAL16V8 registered output; `.CLK`/`.ARST`/`.APRST` on anything but a GAL20RA10), an out-of-range pin or OLMC count, or an over-capacity output (below). Strict validation is a pure accept/reject **gate**: a behavior block that passes it simulates **identically** to the same block in extended mode (FR-079a) — strictness never changes simulation semantics, only whether the block is accepted. The per-OLMC product-term profile (the 22V10's 8-to-16-terms-per-output limit, etc.) is checked only **conservatively**: faithful term counting would require the logic minimization that GALasm performs but this simulator does not, so the check counts product terms as written (and may thus reject a block that real GALasm would fit after minimization) or reports the term-count fit as unverified; the exact, cheap structural limits (single global clock, pin/OLMC counts, suffix and `AR`/`SP` legality) are enforced fully.
- FR-079c: **Buried registered internal nodes.** A 74-series behavior block (FR-079/FR-079a) may reference registered state that is exposed on **no pin**, so parts with hidden internal flip-flops can be modeled faithfully — the immediate case being the 74HC165 8-bit parallel-in/serial-out shift register, whose eight stages surface only the last one (`Q7` and its complement `/Q7`), the other seven being buried. A type declares its buried nodes with an optional top-level **`internal:`** list of node names (§7.6). Each declared node **must** be defined by exactly one registered (`.R`) equation in the behavior block and may be read on any right-hand side (including its own, for a hold/self-feedback term), but it drives no pin and appears on no net. In every other respect a buried node behaves exactly like a registered pin output: it latches its sum-of-products D input on the same clock it would if it had a pin (the global `clock:` pin, FR-062d, or its own `.CLK`, FR-079a); it powers up **U** (FR-079); and it participates in the four-state, selectively-pessimistic, **one-unit-delay** model (FR-077/FR-078) — a buried node's value reaches the equations that read it delayed by exactly one unit, identically to a real net. Consequently a shift chain written over buried nodes (`Q1.R = Q0`, `Q2.R = Q1`, …) advances **exactly one stage per clock**, never all stages at once. Both the slow simulator and the generated C engine (FR-107/FR-109) realize buried nodes identically, so the two engines still agree on every exposed net (FR-107). A declared internal-node name must be a legal signal name (FR-079), distinct from every pin signal and from every reserved word; a declared node with no `.R` equation, a declared node driven by a non-`.R` equation, and a name that collides with a pin or reserved word are each Run-preflight errors reported like the other FR-079 checks. Buried nodes are internal only — they are not probeable by an indicator (FR-068) and never become test-vector columns (FR-115b). (This does not add asynchronous variable load, which remains outside the dialect per FR-079a.)
- FR-080: A 74-series type whose YAML has no behavior block shall leave its outputs at U for the entire run; the simulator shall report each such type once via the message tray (FR-074) when the simulation starts.
- FR-081: At every step the simulator shall evaluate all drivers of every net — including every driver of a tristate net, so that conflicts are always detectable. A net's value is: the common value of its enabled (non-Z) drivers if they agree; Z if it has no enabled driver and no weak driver (FR-083); the conflict result of FR-082 otherwise.
- FR-082: A bus conflict — enabled drivers of one net disagreeing 0-versus-1 — shall set the net to U and be reported visually: all of the net's wire/bus segments render red for as long as the conflict persists, and the message tray (FR-074) names the conflicting drivers (e.g., "bus conflict: U3.Q0 vs U7.B2"). The simulation continues running.
- FR-083: The pull-up and pull-down built-ins (FR-069, FR-070) are weak drivers: they determine the net value (1 for pull-up, 0 for pull-down) only when the net has no enabled strong driver; an enabled strong driver overrides them silently. A pull-up and a pull-down on the same net with no enabled strong driver is a conflict per FR-082. **"Enabled strong driver" means a strong (non-weak) driver contributing a non-Z value — that is, a value of 0, 1, *or* U.** A strong driver whose value is U (for example a tristate output with an uncertain enable, which FR-081 evaluates pessimistically to U rather than Z) is therefore still "enabled": it suppresses every weak driver on the net and the net resolves to U, *not* to the weak pull value. Equivalently, weak drivers decide a net only when **every** strong driver on it is high-impedance (Z); strength dominates the *presence* of a drive, and value (0/1/U) is judged only within the deciding strength tier. (Clarified 2026-06-23: this states explicitly the strong-U-suppresses-weak behavior the resolver has always implemented; no behavior change.)
- FR-083a: **Bidirectional switch elements (dynamic net merging).** The transmission gate (FR-071g) and the relay contact (FR-071h) are **switch elements**. A closed switch does not drive its terminals — it makes the nets attached to its contact terminals **the same electrical net** for that step's resolution. Each unit step: (1) every switch's control — the transmission gate's EN net, the relay's COIL net — is read from the **previous** step's net values, preserving the standard one-unit delay from a control change to the contact change (FR-078); (2) the nets joined by closed contacts are merged **transitively** (a chain of closed switches forms one merged group); (3) each merged group is resolved **once**, over the union of its member nets' driver contributions, using the ordinary net resolution (FR-081–FR-083), and every member net receives the resolved value. Consequences, all required: drive **strength** is preserved across a closed contact (a weak pull seen through a closed relay contact is still weak and loses to a strong driver, FR-083); disagreeing enabled strong drivers on the two sides of a closed switch are a **bus conflict** per FR-082, reported and rendered red on every conductor of the merged group; an indicator or probe on any member net shows the merged group's resolved value. A switch whose control reads **U** (or Z, which reads as U, FR-077) is in an unknown position: it joins nothing, and instead every net in the merged group of each of its contact terminals is forced to **U** for that step — the conservative interpretation. **No charge storage (explicit non-goal):** an open switch isolates its terminals completely, and an isolated net with no driver resolves to Z (FR-081) — the simulator never retains a previous value on an isolated node, so dynamic-latch and precharged-bus techniques that depend on charge retention are not simulated; a design needing retention shall add an explicit weak keeper (pull-up/pull-down, FR-083). Switch feedback whose control depends on nets it merges may oscillate; that is caught by the existing settle bound (FR-085). Test-vector runs (§3.19a) use the slow engine and inherit these semantics unchanged; switch elements contribute no vector columns (FR-115b). **Slow simulator only for now:** the fast engine (§3.23) does not support switch elements — Generate C refuses a design containing one with a clear message (FR-116) — and the NDL export records them as virtual built-ins (FR-119a). Fast-engine support is a possible later additive change; because both engines deliberately share the net-resolution semantics (FR-107), adding it is not expected to rework this FR. (Added 2026-07-07.)
- FR-084: The clock built-in (FR-071, FR-071a) shall output a square wave with 50% duty cycle derived from its effective `period` property: low from t = 0, with the first rising edge after half a period. The simulation shall advance period × speed simulated nanoseconds (unit steps) per real second of wall-clock time, per the effective `speed` property. If multiple clock generators are placed, each toggles per its own `period`, and the pacing rate is the maximum period × speed over all of them.
- FR-085: A design containing no clock generator is purely combinational. It shall be simulated live and continuously — like a sequential design — until stopped via the toolbar button (FR-076); it is no longer run as a single implicit cycle that auto-terminates. On starting, and again after any interactive input change (FR-087b), the simulator runs unit steps as fast as practical (unpaced) until the circuit reaches quiescence (no net changes value between consecutive steps), displays the results, and then idles — performing no further steps and consuming no CPU — until the next interactive input perturbs it, whereupon it re-settles. If a settling episode does not reach quiescence within a bound of 10,000 units, the simulator shall report this once via the message tray (FR-074) — a likely oscillation — and pause evaluation (idle) without terminating the run; the user may press Stop. Indicators reflect the live simulated state throughout, and on Stop the last values remain displayed until the design is next modified. (Reworked 2026-06-13; supersedes the single-implicit-cycle, auto-terminating combinational model, which left interactive inputs (FR-087b) useless because the run ended before the user could act.)
- FR-086: A design containing at least one clock generator is sequential and shall run, paced per FR-084, until stopped via the toolbar button (FR-076).
- FR-087: While the simulation is running, the design shall be read-only and the editor's selection shall be locked: placing, wiring, moving, rotating, deleting, property/override editing, undo/redo, paste (FR-112), New, Open, and changing the selection shall be disabled; pan, zoom, Save, and Save As shall remain available (saving a running design is harmless and deliberately permitted — Save As included, clarified 2026-07-08). A click that would otherwise select an item — a wire, a bus, or a non-interactive component — shall instead post a status-bar message (FR-074) reading "Editor is locked while the simulator is running" and leave the current selection unchanged; a click on empty canvas is ignored silently (no message, no rubber-band). The lone exception is a click on an interactive built-in (FR-087a/FR-087b), which applies its input action rather than reporting the lock. On entering the simulation the current selection shall be cleared (it cannot be changed while locked). (Reworked 2026-06-17; supersedes leaving selection available during a run, which let the user retarget the selection and clear it but served no purpose while editing was locked.)
- FR-087a: The input switch built-in (FR-071c) is a strong driver: at every step it shall drive its output to the logic value of its current state — 1 or 0 (never U or Z), overriding any weak pull-up/pull-down driver on the net (FR-083). While a simulation is running (FR-076), clicking the switch shall toggle it to the other state (0↔1), changing its driven output and re-evaluating the simulation. The click is an interactive input (FR-087b): a deliberate exception to the read-only rule (FR-087) that updates the switch's persisted state (FR-071c) and marks the design modified (FR-049a, FR-092) but is not placed on the undo stack (undo/redo remain disabled during simulation, FR-087). While not simulating the switch is not a click target: a click selects it as an ordinary component (FR-016), and its state is set through the properties panel (FR-020c). (Reworked 2026-06-17; supersedes the three-position ? → 1 → 0 → ? dial cycle and the U-driving ? position.)
- FR-087b: The simulator shall provide a general interactive-input mechanism for built-in objects that accept manual user input while a simulation runs. A built-in type may register an interaction handler — the input-side analogue of its behavior (FR-067a) — that mutates the instance's interactive state. While simulating, a click on such an object's body shall invoke its handler as a live, non-undoable design change (FR-087a) and shall cause the simulator to re-evaluate from the changed state (re-settling a combinational design per FR-085, or taking effect on the next step of a paced design per FR-086). The mechanism is general and not specific to any one object: a new interactive input is added solely by registering its handler (and render). The input switch (FR-071c/FR-087a) is the first interactive input object; clicks on non-interactive objects are caught by the selection lock (FR-087), which reports "Editor is locked while the simulator is running" instead of selecting.

### 3.19a Test Vectors

- FR-115: The application shall provide **test vectors**: an authored table of input patterns paired with expected outputs, run against the slow (debug) simulator (FR-075/§3.19) to score a design's behavior automatically, as an alternative to manually toggling input switches (FR-071c) and reading indicators (FR-068/FR-071d). The feature is reached from the top bar (FR-004a). It covers **both** combinational designs — no clock generator (FR-085), each row evaluated independently per FR-115c — and **sequential (clocked) designs**, whose rows run in order with scripted clocks per FR-115e. (Reworked 2026-07-01; supersedes the combinational-only first version, which deferred clocked designs to a later extension.) The input stimulus drives the design's input-switch built-ins and the observed outputs are its indicator built-ins (FR-115b); in addition the design's **ports** (FR-094 / multi-bit portN, FR-071e) are bound as inputs or outputs by their effective direction (FR-115f). (Added 2026-06-26.)
- FR-115a: A test-vector set is stored in a **sibling file** — a JSON document with the extension **`.tv`**, conventionally saved beside the design file it tests. It carries a numeric `formatVersion` (migrated forward like the design save format, FR-060c), the ordered **input columns** and **output columns** — each identifying a connection point by the instance reference designator (`refdes`, FR-011/FR-011a) and pin name, with a cached display label — and the **rows** (each an array of input symbols and an array of expected-output symbols). Columns bind by `refdes`+pin, not by label, so relabeling a switch or indicator (FR-011b) never invalidates a file. The file is loaded and saved through the **server-side file-navigation dialog** (FR-053, the same browser used by Open and the ROM picker, FR-114a), defaulting to the directory of the current design with a name derived from it. On load, the file's columns are reconciled against the design's actual bound columns — switches, indicators, ports (FR-115f), and clock generators (FR-115e) — by `(refdes,pin)`; a column present in the file but absent from the design (or vice-versa) is surfaced as a non-fatal warning rather than rejecting the file.
- FR-115b: The test-vector **table editor** is a **docked panel** occupying the bottom third of the canvas area (design §6.16), leaving the schematic visible and navigable (pan/zoom) above it; it is **modeless** and is toggled open and closed from Simulate ▸ Test Vectors (FR-004a). (Reworked 2026-07-02; supersedes the former modal dialog — the same modal idiom as the New GAL part / New memory device dialogs, FR-066c/FR-114 — which covered the schematic the user needs to see while authoring vectors. While the panel is open the design is read-only, FR-115h. The `.tv` Load/Save file browser, FR-115a/FR-053, remains a transient modal launched from the panel.) Its columns are **auto-derived from the open design**: one **input** column per input-switch built-in (FR-071c), whose cells take the symbols `0` or `1`; for a clocked design, one **input** column per clock generator (FR-071), whose cells take `0`, `1`, or `C` (FR-115e); and one **output** column per indicator bit — a 1-wide indicator (FR-068) contributes one column, an 8-wide indicator (FR-071d) contributes eight columns `D0`…`D7` — whose cells take the symbols `H` (expect logic 1), `L` (expect logic 0), or `X` (don't-test); plus the design's port-bound columns (FR-115f), unioned with these. Columns are ordered by reference designator and labeled by each instance's display label (FR-011b), falling back to its refdes. The panel lets the user add and remove rows, **Load** and **Save** a `.tv` file (FR-115a), **Run** the vectors (FR-115c), and **Capture** expected values (fill the expected-output cells of every row from a simulator run of that row's inputs, FR-115c) — Capture authors a golden table from the circuit's current behavior, which the user may then edit. The Test Vectors command and the panel's Run/Capture actions are unavailable while the interactive simulator is running (FR-087); a vector run is independent of, and does not start, the interactive simulator (FR-076).
- FR-115c: **Combinational run semantics.** These semantics govern a design with **no clock generator**; a clocked design runs per FR-115e instead. Running a vector set evaluates each row **independently**: the simulator drives every input switch to the row's input symbol **and forces every bound input-port net to it** (via the simulator's external stimulus, FR-115f), settles the design to quiescence exactly as the live combinational simulator does (FR-085, including its 10,000-unit oscillation bound), then reads each output column's net value (FR-077) — a port output is read directly from its own net (FR-094e). A cell passes when `H` matches logic 1, `L` matches logic 0, or the expected symbol is `X` (always passes); the four-state values U and Z never match `H` or `L`. Capture (FR-115b) records the settled value of each output as `H` (logic 1) or `L` (logic 0) — a U/Z result is recorded as `X` (no assertion). A vector run executes on a **throwaway copy** of the design: it never mutates the live design, never marks it modified (FR-049a/FR-092), and never touches the undo history (FR-024). If the design contains memory devices (FR-114), their ROM contents are loaded for the run exactly as the interactive simulator loads them (FR-114e).
- FR-115d: **Results.** After a run the panel (FR-115b) shows, per output cell, whether it **passed** or **failed**, a failing cell additionally showing the actual simulated value; and a summary of how many of the total rows passed (a row passes when all of its non-`X` output cells pass). Results are presentational and are not persisted in the `.tv` file (FR-115a), which records only inputs and expected outputs.
- FR-115e: **Sequential run semantics.** A design containing at least one clock generator (FR-071 — the condition that makes it sequential, FR-086) is tested with **time-ordered** vectors: the rows run **in order on a single simulation instance**, so register and net state persist from each row to the next, and the run **owns the clock deterministically** — the time-based behaviors of the clock and power-on-reset built-ins (FR-084/FR-071b) are suppressed for the run and their nets driven by the runner as scripted stimulus, in place of the interactive simulator's wall-clock pacing (FR-084/FR-086). Specifics:
  - **Clock columns.** Each clock generator contributes an **input column** — identified by the clock's own `(refdes, "OUT")`, like every other column (FR-115a) — whose cells take `0`, `1`, or `C`, in the style of classic PLD/JEDEC test vectors: `0`/`1` hold the clock net at that level for the row; `C` applies one full positive pulse. A newly added row defaults its clock cells to `C` (one row = one clock cycle) and its other input cells to `0`.
  - **Row semantics.** Each row, in order: apply the row's input symbols — switch states, port stimulus (FR-115f), and clock levels (`C` applies low) — and settle to quiescence exactly as FR-085 (each settling phase bounded by the same 10,000-unit limit); then, when the row has `C` cells, pulse those clocks **together** — drive high, settle, drive low, settle; then read each output column's net and score it by FR-115c's matching rules (`H`↔1, `L`↔0, `X` always passes; U/Z match nothing). A `0`→`1` clock level change between consecutive rows is a rising edge like any other, so half-cycle stimulus can be scripted with level cells.
  - **Power-on preamble.** Before row 1, the run executes an implicit reset phase mirroring the interactive simulator's power-on (FR-071b): every power-on-reset built-in drives R=1 (/R=0) while the runner applies scripted clock pulses — each reset instance releasing (R=0, /R=1) after its own `cycles` property's worth of pulses, the phase lasting the maximum `cycles` over the instances — followed by a final settle. The preamble is not visible as a row; a clocked design with no reset built-in has no preamble. Registers still power up U (FR-079), and every clock net starts driven `0` (the FR-084 t=0 level).
  - **Capture** (FR-115b) performs the same ordered pass — preamble, then each row's inputs and pulses — recording each row's settled outputs in sequence.
  - A design with **no** clock generator keeps the independent-row combinational semantics (FR-115c) unchanged; the `.tv` file shape is common to both (FR-115a, design §7.7 — the `C` symbol is legal only in a clock column).
  - **Hierarchy (FR-102/FR-103).** A vector run flattens the design exactly as the interactive Run does, so embedded sub-designs and peer sheets participate; columns still bind only to the top sheet's switches/indicators/ports (FR-115b/FR-115f). **Deferred scope:** a clock generator *inside* a flattened child or peer sheet cannot be a scripted column, so the runner refuses such a design with a clear message rather than letting the unscripted clock float U. (Added 2026-07-04.)
  (Reworked 2026-07-01; supersedes the deferred stub recorded 2026-06-26, with two refinements to its intended direction: rows advance by **settle-to-quiescence phases** rather than the stub's "defined number of steps per row" — a fixed step count is fragile against logic depth, while per-phase quiescence matches FR-085 — and power-on reset is handled by the implicit preamble above rather than scripted reset columns, keeping the table small and matching the interactive simulator's power-on feel.)
 - FR-115f: **Binding to ports.** Beyond input switches and indicators (FR-115b), a design's **ports** (FR-094 and the multi-bit portN, FR-071e) are bound as test-vector columns by their **effective direction** (FR-094c/FR-094d): a port (or port bit) whose effective direction is **in** becomes an **input** column; one whose effective direction is **out** becomes an **output** column. A multi-bit portN of width N contributes **N one-bit columns**, one per bit, uniformly — there is no whole-bus column, and a 1-wide port is simply the N=1 case. A port whose effective direction is **bidir** (a genuine bidirectional net carrying no override, FR-094d) cannot be a single input-or-output column and is **omitted** with a non-fatal warning, exactly as an unmatched column is (FR-115a); setting that port's direction override (FR-094d) is how the user opts it into testing. The unit under test is **always the design currently shown in the editor** — there is no separate "pick a child file" step. A port column is identified, in the table and the `.tv` file (FR-115a), by the **port's own** reference designator and pin — `(refdes, "P")` for a 1-wide port, `(refdes, "P"i)` for a portN bit — the natural, stable identity that reconciles by `(refdes,pin)` across sessions; **no helper components are synthesized**. The run drives and reads ports directly: an **input** port is driven by forcing its net through the simulator's external stimulus (FR-115f relies on a port pin being a net member, FR-094e), reusing the design's combinational run (FR-115c) unchanged; an **output** port is read straight from its own net (FR-094e). Port-derived columns **coexist** with any input-switch/indicator columns the design also contains (FR-115b) — the two sets are unioned. (Added 2026-06-30; implements the port binding deferred at FR-115's introduction.)
- FR-115g: **Superseded — clocked-design guard (removed).** While test vectors were combinational-only, the table editor detected a design containing a clock generator and disabled its Run and Capture actions with a notice (the runner would otherwise have settled each row of a never-quiescing clocked design to the full 10,000-unit bound with no warning). Sequential designs are now first-class (FR-115e), so the guard is removed: the dialog runs and captures clocked designs normally, and no disable or notice applies. (Added 2026-07-01; superseded later the same day by the FR-115e sequential implementation. Retained for FR-number history.)
- FR-115h: While the test-vector panel (FR-115b) is open the design shall be **read-only**, generalizing the interactive-simulator lock (FR-087) so the two share one condition — the design cannot be edited while *either* a simulation is running *or* the panel is open. The panel's lock behaves exactly like the simulation lock, minus the live-sim-only click-to-toggle-a-switch exception (FR-087a), which requires a running simulation: the canvas accepts **panning and zooming only** — no drags, wire/bus starts, placement, or selection — so the user can inspect the schematic behind the reduced canvas, and a click on a selectable item posts the read-only notice rather than selecting it. Concretely: Wire/Bus, Undo/Redo, New/Open/Refresh Types, Paste, and all canvas mutations (move, delete, rotate, wire/bus draw, property edits) are disabled, while the zoom/Fit commands, Save, and Save As stay enabled and the Select tool button stays lit (as while simulating, FR-087). Copy stays read-only (FR-111) but is moot, since nothing can be selected under the lock. The panel and the interactive simulator are **mutually exclusive**: the interactive **Run** button is disabled while the panel is open (the reciprocal of FR-115b's rule that Test Vectors is disabled while simulating), so a vector run and a live run can never overlap. The Simulate ▸ Test Vectors item itself stays enabled while the panel is open, so it can toggle the panel closed; closing the panel restores full editing. Because the design cannot change while the panel is open, the panel's auto-derived columns (FR-115b) are a stable snapshot for its lifetime and need no live refresh. (Added 2026-07-02.)

### 3.20 Type Data Refresh

- FR-088: The application shall provide a "Refresh Types" action (File menu, FR-004a) that re-copies type data from the currently loaded component library into the placed instances of the open design, so edited component definitions reach existing instances without deleting and re-placing them (placement-time copies otherwise persist indefinitely per FR-057). For each instance whose type id (FR-066e) exists in the library — built-ins refresh from the client registry (FR-067) — the copied type data is replaced while preserving position, rotation, reference designator, wiring, and per-instance overrides (FR-058); an override key that no longer matches any delay or declared property in the new definition is dropped. A memory instance's **content binding** — its `mem.romFile` (FR-114e) — is per-instance state that rides in the copied type data: a refresh shall preserve the instance's existing `romFile` rather than adopt the library metatype's creation-time path. (Reworked 2026-07-06; previously the wholesale type-data copy silently rebound a refreshed ROM to the metatype's file — found when a batch refresh broke the rom-demo parity example.) An instance shall be skipped — left unchanged and reported once per type via the message tray (FR-074) — when the new definition is structurally incompatible: its render type changed, or a pin currently referenced by a wire or bus connection no longer exists (for subunit packages: no longer exists in the same unit). The whole action is one undoable operation (FR-024) and is unavailable while simulating (FR-087). The action reads the client's already-loaded library: picking up *external* YAML edits still requires a server restart and page reload first (FR-007); in-app *creation* of a new part is the one exception that goes live without restart (FR-007a).

### 3.21 Server Connection Resilience

- FR-089: The client shall monitor server availability by polling a lightweight health endpoint (GET `/api/v1/ping`) approximately every 3 seconds. A connection tray at the right end of the status bar (FR-072) shall show the current connection state ("connected" / "disconnected").
- FR-090: When the server becomes unreachable, the application shall keep all client-side editing fully available (the design's source of truth is the browser store), show "disconnected" in the connection tray, and post instructions via the message tray (FR-074): the user's work is retained in the browser tab, the page must not be reloaded, and the server should be restarted at the same address and port (the client can only reach the origin that served it, NFR-002). Server-dependent operations (Save, Open, directory listing) shall fail with a clear message and without corrupting or discarding client state.
- FR-091: When polling detects the server again — a fresh instance at the same origin suffices, since the server is stateless and every save transmits the complete design (FR-046) — the application shall resume automatically: show "connected", post a reconnection message, and, if the design has unsaved changes, save it immediately to its known save path and report the result, or open the Save dialog if it has never been saved.
- FR-092: The application shall maintain a local backup snapshot of the working design in browser localStorage: updated (debounced) after each design-modifying action while unsaved changes exist, recording the serialized design (FR-055 shape), its save path, design name, and a timestamp; removed on successful save. This protects against the loss modes reconnection cannot cover: page reload, tab close, and browser crash.
- FR-093: At startup, if a backup snapshot exists (a previous session ended with unsaved changes), the application shall offer recovery before presenting the default empty design (FR-045): accepting restores the snapshot's design, name, and save path with unsaved-changes status set; declining discards the snapshot.

### 3.22 Sub-Designs, Ports, and Off-Sheet Connectors

This group adds hierarchical design: a separately-saved design may be embedded in a higher-level design as a single component (a **sub-design**), and a circuit may be split across peer sheets joined by **off-sheet connectors**. It builds on the first-class vertex connectivity model (design.md §7.1a), which reserved a `connector` vertex kind for this purpose. The two mechanisms are distinct features sharing one primitive — the **port** (FR-094): embedding *encapsulates* a child as a component (FR-098); off-sheet connectors join peer sheets *flatly* (FR-101).

**Ports (the interface primitive)**
- FR-094: The editor shall provide a built-in **port** object (FR-067), placed from the lower palette region (FR-006a). A port has a user-set signal **label** and a **direction** (in/out/bidir) that is **derived from the port's wiring** (FR-094c) rather than set by hand. It exposes a single **one-bit** connection point and marks the net attached there as part of the design's external interface (FR-095). A pin is always one bit; bit-width belongs to buses and pin groups, never to a pin or a single port — for a multi-bit (bus) interface, use the multi-bit port (FR-071e). Ports are assigned A-series designators (FR-011a) and are otherwise ordinary instances (select/move/rotate/delete/persist/wire). A port's connection point is a `connector` vertex (design.md §7.1a). (Reworked 2026-06-27; supersedes the per-port `width` field — a single connection point carries one bit, so a width>1 setting on a 1-wide port was the long-standing "fat pin" confusion. Multi-bit interfaces are now expressed solely by portN, FR-071e.)
- FR-094a: Within one design, all 1-wide ports sharing the same label denote the same net (connectivity by label), so an interface net may appear at several points on the sheet without a drawn wire between them. Label match is exact. (Multi-bit ports, FR-071e, do not participate in label joining — their pins join nets only through a drawn bus/wire.)
- FR-094c: A port's **direction** shall be **derived from the net its connection point joins**, tracking the wiring live rather than being set manually. Examining the non-port pins on that net (within the design; same-label ports, FR-094a, share the net): the direction is **bidir** if any such pin is itself bidirectional or three-state (e.g., a RAM/ROM data line — any pin whose own direction is `bidir` or `tristate`); else **output** if the net is driven by a plain output pin; else **input** (the default, also for an unconnected port). The properties panel (FR-094/FR-020) displays this derived direction **read-only** for a port whose derivation is definite (`in` or `out`); only the genuinely ambiguous **bidir** case is user-settable, via the override of FR-094d. Because the value is derived and ports join nets by label (FR-094a), the netlist and simulator are unaffected; direction feeds the sub-design IC interface layout (FR-099), the test-vector port binding (FR-115f), and the display. The direction written to the save format (FR-060b) is the **effective** value (the override applied where present, FR-094d), recomputed wherever it is used. (Added 2026-06-27; supersedes the manual in/out/bidir selector that previously let a port's direction disagree with its wiring — e.g., an output port left at the `in` default rendered on the wrong side of the embedded IC. The FR-094d override re-admits manual control **only** for the bidir case, where the wiring is genuinely ambiguous, so that supersession is preserved for definite directions.)
- FR-094d: When a port's **derived** direction (FR-094c) is **bidir** — the conservative result whenever its net touches any bidirectional or three-state pin, which by itself cannot distinguish a true bidirectional bus from a three-state **output** acting as a switchable driver (e.g. a 74381's three-state F outputs feeding a multiplexed bus) — the user may set an optional **direction override** of `in` or `out`. A port's **effective direction** is its override when one is set *and* the derived direction is `bidir`; otherwise it is the derived direction itself. The override therefore takes effect only in the genuinely ambiguous bidir case: a port whose wiring yields a definite `in` or `out` ignores any override and stays fully derived (read-only), preserving FR-094c's guarantee that a definite direction can never disagree with the wiring; and if the wiring later changes so the derivation is no longer bidir, the override lies dormant (the derived value wins) and never goes stale. The override applies equally to a 1-wide port (FR-094) and to a multi-bit portN (FR-071e, whose direction is aggregated across its bits — the override sets the whole group). The **effective** direction is what feeds every consumer of port direction: the embedded sub-design IC interface layout (FR-099), the test-vector port binding (FR-115f), and the display; the save format (FR-060b) records the effective direction **plus the override itself**, so the intent survives reload. The properties panel exposes an editable in/out/bidir selector (defaulting to bidir, writing the override and clearing it when set back to bidir) **only** for a derived-bidir port; for a definite-direction port the panel stays read-only (FR-094c). (Added 2026-06-30.)
- FR-094b: A port is drawn on the canvas as a **pentagon** ("flag"): a rectangular body carrying the label, one edge tapering to an **apex**. The apex points **off-sheet** (the "front"); the connection pin sits on the opposite, flat edge facing **into the sheet** (the "back"). The apex↔pin relationship is fixed to the part, so it is preserved as the instance is rotated (FR-020); the label stays upright regardless (FR-015). This is the 1-wide analogue of the 8-wide port's pentagons (FR-071e). (Supersedes the earlier box rendering.)
- FR-094e: A port's connection point shall participate in the netlist as a **member of the net it taps**, so that net is both **observable** and **drivable** by the port's reference designator and pin — `(refdes, "P")` for a 1-wide port, `(refdes, "P"i)` for a multi-bit portN bit. This makes a port a first-class probe/stimulus point for the simulator: its net can be read (`valueOfPin`) and driven by external stimulus (FR-115f), without changing connectivity or the port's own electrical behavior (a port still drives nothing of its own). (Added 2026-06-30; previously a 1-wide port's connector contributed only to label-based net joining (FR-094a) and was invisible to per-pin queries — a portN's bit pins were already net members via their bus group-snap.)

**Interface and default rendering**
- FR-095: A design's **external interface** is the set of its ports: each distinct **1-wide** port label (FR-094) contributes one **one-bit** interface pin; each multi-bit port (FR-071e) contributes a **pin group** of N one-bit pins (named `<label>0`..`<label>(N-1)`, group named `<label>`) to which a matching-width bus snap-connects via the normal pin-group machinery (FR-041/FR-042). Every interface pin's direction is the port's **effective** direction (derived, FR-094c, or overridden for a bidir port, FR-094d). A multi-bit interface is therefore a pin group, not a single "wide" pin — pins are always one bit. A design with no ports has no interface and cannot be embedded (FR-097a). (Reworked 2026-06-27; supersedes "one pin per label carrying that label's width".)
- FR-096: A design may carry a design-level **default embed rendering** — `ic` (integrated-circuit rectangle) or `connector` (connector strip) — settable while editing that design and persisted in its file. A design that has never set one defaults to `ic`.

**Embedding (the ADD flow)**
- FR-097: The lower palette region (FR-006a) shall include a single **ADD** entry. Placing it on the canvas (drag or click-to-place, FR-008/FR-009) opens an "Add sub-component" dialog instead of immediately creating an object; placement remains one-shot (FR-010), returning to select-tool mode whether the dialog is confirmed or cancelled.
- FR-097a: The dialog shall let the user (a) choose a sub-design file via the file-navigation mechanism (FR-052/FR-053), (b) view the chosen file's default rendering (FR-096) and its derived interface (FR-095), and (c) override the rendering (`ic` vs `connector`) for this placement. Confirming creates a sub-design instance (FR-098) at the placement point; cancelling places nothing. A file with no ports (no interface, FR-095) shall be reported and shall not create an instance; a file that would embed itself or form an embedding cycle (FR-102a) shall be rejected with a message.
- FR-097b: A sub-design instance's child reference is held in memory as an **absolute** path while editing and is written to the parent file as a path **relative to the parent's save directory** (FR-098), the relative form computed only at save time when the parent's directory is known. Embedding therefore does **not** require the parent to be saved first: the user chooses the child file and the instance is placed immediately, like any other component (FR-008/FR-009), with no intervening save prompt. (Reworked 2026-06-27; supersedes the prior rule that embedding required saving the parent first and prompted to save before completing the embed — the relative path is only needed on disk, not to place the instance. Returning from a *descended* child to an unsaved parent is handled separately, FR-100a.)

**Sub-design instances**
- FR-098: A **sub-design instance** references its child design file. **On disk** the reference is stored as a path **relative to the parent's save directory** (FR-060b), resolved to an absolute path when the parent is opened. **In memory** the editor holds the resolved **absolute** path, so an instance can be embedded before the parent has a save location (FR-097b); the absolute↔relative conversion happens at the save/load boundary. The instance is a live reference: it stores no copy of the child's contents or interface. (This **supersedes FR-057** for sub-design instances, which carry a child path and a chosen render style in place of copied `typeData`.) The child's interface (FR-095) is re-derived from the child file on load. (In-memory absolute path added 2026-06-27.)
- FR-098a: Sub-design instances are assigned reference designators from a separate **X-series** (X1, X2, …), incremented from the highest existing X-number in the parent (cf. FR-011/FR-011a; they consume neither U- nor A-numbers). The same child file may be embedded multiple times in one parent; each placement is an independent instance.
- FR-099: A sub-design instance is rendered in its chosen style (FR-097a). `ic`: a rectangle with interface pins on the left (inputs, and bidir) and right (outputs), by each port's effective direction (FR-094c/FR-094d), labeled by port label. `connector`: a tall, narrow rectangle with all interface pins ranked along one long edge, labeled by port label, ordered by label. In both styles the pins are the child's interface (FR-095); pin and designator labels render upright (FR-012/FR-015); the child's internals are not drawn (encapsulated). The render style is **purely cosmetic** and does not change electrical or simulation semantics. A multi-bit interface signal (from a portN, FR-071e) appears as a **pin group of N one-bit pins** on the appropriate side, contiguous and in bit order, to which a matching-width bus snap-connects via the normal pin-group machinery (FR-041/FR-042); a narrower bus may take a free sub-block (FR-041c).
- FR-099a: If a sub-design instance's child file cannot be resolved or loaded when the parent opens, the instance shall render as a distinctly-marked **broken-link placeholder** naming the missing relative path, and the condition shall be reported once via the message tray (FR-074); it shall not abort loading or crash the editor (consistent with the renderer's unknown-type placeholder, design.md §6.8).
- FR-099b: Because the reference is live (FR-098), a child interface change reaches existing instances on the next load of the parent. A parent connection to an interface pin whose port label no longer exists in the child shall be left dangling (FR-029/FR-030) and reported via the message tray (FR-074).
- FR-099c: When loading a parent reveals that an embedded child's **interface changed** since the parent was last saved — detected by comparing the freshly resolved interface (FR-095) against a compact per-instance record of it persisted with the parent (design §7.2; the only exception to FR-098's store-nothing rule, kept solely for this comparison, never trusted for rendering or simulation) — the parent's **simple** wires attached to that instance (a wire with exactly two endpoints and no junction or tap along it) shall be automatically **re-routed** with the standard wire auto-router, replacing their now-stale interior bends, and the condition reported per instance via the message tray (FR-074). Complex wires and dangling ends (FR-099b) are left for manual cleanup, and a wire the router cannot route keeps its old bends. Re-routing changes geometry only — never connectivity — and, like FR-099b's dangling rewrite, is load-time normalization: not a command, not undoable, and not marking the design modified. A parent saved before this record existed lacks it: no re-route occurs on its first load, and the record is written at its next save. (Added 2026-07-04.)

**Navigation between sheets**
- FR-100: The user shall be able to descend into a sub-design instance (double-click, or a context-menu "Open sub-design", FR-033b) and to follow an off-sheet connector (FR-101); either replaces the editing canvas with the target design file. Such a navigation is treated as closing the current design: the unsaved-changes warning (FR-049a) applies, so the user must save or knowingly discard before the canvas changes.
- FR-100a: The editor shall maintain a navigation **back-stack** (breadcrumb) recording the chain of sheets descended through, so the user can return to the parent and up the chain. Returning re-opens the recorded **file** and is itself a save-or-lose navigation (FR-100). Because returning re-opens the parent from its file, descending into a sub-design while the parent is unsaved shall first prompt to save the parent (FR-047) so there is a file to return to; declining cancels the descent. (Interim 2026-06-27: a future change may retain an unsaved parent in memory so even descent needs no save; until then only descent — not embedding, FR-097b — prompts to save.)

**Off-sheet connectors (peer sheets)**
- FR-101: A port (FR-094) may additionally carry an **off-sheet target**: a **sibling design file in the same folder** (a bare filename, no directory path) and a port label within it. All sheets of one circuit therefore live in a single folder — off-sheet connectors never cross directories (this deliberately excludes the relative-path form that FR-098 uses for sub-design embedding, which is a distinct mechanism). A port with a target is an **off-sheet connector**, marked by a small filled triangle at its pentagon's apex; **double-clicking** it — or choosing **"Follow off-sheet connector"** from its context menu (FR-033b) — navigates to the target file (FR-100), while a plain click still just selects the port (FR-016). Off-sheet connectors join peer sheets of one circuit (a flat relationship), distinct from the encapsulating embedding of sub-design instances (FR-098). (Reworked 2026-07-08: the vague "clicking/opening it navigates" became the double-click/context-menu gesture, matching the sub-design idiom of FR-100 so plain selection is preserved. Same-folder rule added 2026-07-08, superseding the earlier "relative path, per FR-098": a directory path in the target confused the flat peer relationship with hierarchy — peer sheets are now constrained to one folder and the target carries only a bare filename.)
- FR-101b: The properties panel (FR-020a), when the single selected component is a **1-wide port** (FR-094), shall present two editable **off-sheet target** fields: the target **file** (a **bare design-file name in the same folder** as the current design — no directory path, FR-101) and the target **label** (the port label within that file, FR-101a). Entering a non-empty file creates or updates the target; clearing the file removes it (the label field alone never creates a target). A file value containing a path separator is **rejected** (the field is reverted to its prior value and no target is written), enforcing the same-folder rule (FR-101). Edits are undoable (FR-024) and are disabled under the simulation and test-vector locks (FR-087/FR-115h) like other property edits. A multi-bit portN (FR-071e) offers no target fields (off-sheet links remain deferred for it). Because resolving the sibling filename — and returning to the referring sheet via **back** — needs the design to have a folder of its own, **following** a connector from a never-saved design shall first prompt to save it — the same interim rule as descending into a child (FR-100a) — and the navigation joins the back-stack (FR-100a) so **back** returns to the referring sheet. (Added 2026-07-08: FR-101 specified the target and the simulator honored it, FR-103, but no UI existed to set or follow one — the gap surfaced by the 2026-07-08 user-manual review. Same-folder rule and path-separator rejection added 2026-07-08 with FR-101's rework — the field now takes a bare filename, not a relative path.)
- FR-101a: An off-sheet connector continues its net to the named port in the named target file (matched by the named label; a port is one-bit, FR-094, so there is no per-bit case — the former "applied per bit for width>1" clause was removed 2026-07-08 with per-port width long gone, and portN defers off-sheet links entirely, FR-071e). This cross-file continuation is established **only** by an explicit target (FR-101) — coincidentally equal labels in unrelated files do not join. The two joined ports denote one net across the two files.

**Simulation of multi-sheet / hierarchical designs**
- FR-102: The slow simulator (§3.19) shall **flatten** hierarchical sub-design instances when evaluating: each instance is replaced by its child's contents, binding the child's interface ports (FR-095) to the nets wired to the corresponding instance pins. Flattening recurses through nested sub-designs. Reference designators are made unique by prefixing each level with its instance path (e.g., `X1/U3`, `X1/X2/U5`); the hierarchical names are used in bus-conflict and indicator messages (FR-082). Port **labels** are namespaced the same way (`X1/CLK`): a label unions nets (FR-094a) only within its own sheet instance, so same-label ports in two different instances — or in an instance and its parent — never join by coincidental label equality (the FR-101a rule applied within the hierarchy). (Clarified 2026-07-04: label namespacing made explicit.)
- FR-102a: The simulator (and the embed dialog, FR-097a) shall detect cycles in the sub-design **embedding** graph — including the mixed case where an off-sheet target (FR-101) leads back into a file currently being expanded as an ancestor embed — and on a cycle shall refuse to run/embed with a clear message via the message tray (FR-074) rather than recursing without bound. Circular references among off-sheet **peer** sheets (e.g. sheet A targets a port on B while B targets one on A) are legal and expected: peer loading de-duplicates by absolute path (FR-103), so they cannot recurse. (Reworked 2026-07-04; supersedes the earlier rule that refused cycles in the off-sheet connector graph as such — mutual peering is how a multi-sheet circuit is normally linked.)
- FR-103: When a design uses off-sheet connectors (FR-101), the simulator shall load the connected peer sheets (following targets transitively, de-duplicating by absolute path, subject to FR-102a) and union nets across files by their declared connector links (FR-101a), so a circuit spread across peer sheets simulates as one. Reference designators (and port labels, per FR-102's namespacing rule) across distinct peer sheets are made unique by prefixing with a per-sheet identifier: the sheet's file base name (e.g. `alu/U3` for a peer loaded from `alu.json`), disambiguated with a numeric suffix on a base-name collision; the root sheet is unprefixed. (Clarified 2026-07-04: per-sheet identifier and label namespacing made concrete.)

### 3.23 Fast (Generated C) Simulator

The "fast" engine of the vision statement (`sim-vision.md`), complementary to the slow simulator (§3.19). It is a **code generator** rather than a second interpreter: it emits a standalone C program that, when compiled, simulates the one design it was generated from. (Supersedes the earlier "fast engine out of scope"; see §1, §7.)

- FR-106: The application shall provide a fast simulator in the form of a code generator that produces, from the current design, a self-contained C program implementing the simulation of that design. The C program is a standalone simulator for that one design; generating it does not require or run the slow simulator (§3.19).
- FR-107: The generated C simulator shall represent exactly the same four signal values as the slow simulator (FR-077) — logic 0, logic 1, U (undefined), Z (high impedance) — using the same selectively-pessimistic combination rules, so that for any given design the two engines agree on every net's value. The fast engine is an optimization of the slow engine, not a different semantics.
- FR-108: The generated C simulator shall evaluate every driver of every net — including every driver of a tristate net — and detect bus conflicts exactly as the slow simulator does (FR-081/FR-082): two or more enabled drivers disagreeing 0-versus-1 set the net to U, and the conflict shall be reported, naming the conflicting drivers (e.g. "U3.Q0 vs U7.B2") consistent with FR-082. (The concrete reporting channel of the standalone program — see OQ-011 — is a design detail; that conflicts are detected and reported is the requirement.)
- FR-109: The code generator shall be implemented in JavaScript and run **in the browser front end**, reachable from the editor (e.g. a toolbar action), not in the Go server. It shall **reuse** the slow simulator's existing JavaScript modules — the single GALasm behavior compiler (FR-079) and the netlist/connectivity builder (FR-034b/FR-059a) — rather than reimplementing them, so the system has exactly one GALasm compiler and one connectivity extractor shared by both engines. This is a deliberate deviation from `sim-vision.md`'s standalone-Go, server-side transpiler: building the generator in JS eliminates a duplicate GALasm compiler and the correctness risk of keeping two compilers in agreement; the vision permits a non-Go choice for a strong engineering reason, and DRY reuse of the slow simulator's parser and netlist code is that reason.
- FR-110: The generated C simulator shall preserve the slow simulator's timing model: unit-delay, double-buffered evaluation with 1 unit = 1 simulated nanosecond (FR-078), so outputs respond exactly one unit after inputs and the two engines settle in the same deterministic sequence. (Batch driving and bounding are now specified: stimulus and run control per FR-117/FR-117a, output per FR-118, runtime-read ROM contents per FR-117b; OQ-012 is fully resolved.)
- FR-116: The code generator (FR-106/FR-109) shall be reachable from a **Simulate ▸ Generate C…** menu item (FR-004a). Choosing it opens the server-side file-save dialog (FR-053, as the `.tv` file does, FR-115a), defaulting to the current design's directory with a name derived from the design, and writes into the chosen directory the generated **`<design>.c`** plus verbatim copies of the fixed runtime pair **`runtime.c`/`runtime.h`** (FR-116a), so the output directory is self-contained and compiles with plain `cc <design>.c runtime.c` — no include paths, extra libraries, or special flags. Generation is read-only with respect to the design (it never mutates it, marks it modified, or touches undo) and neither requires nor runs the slow simulator (FR-106). The menu item is disabled while a simulation runs (FR-087) and while the test-vector panel is open (FR-115h). **Hierarchy:** generation **flattens** the design first — the same flatten the slow simulator uses (FR-102/FR-102a/FR-103) — so hierarchical and multi-sheet designs generate; a flattening refusal (embedding cycle, unloadable child or peer, unresolvable link) fails generation with the same clear message. The baked vector/observable columns (FR-117/FR-118) derive from the **top sheet only**, exactly as the panel's (FR-115e hierarchy rule). Because one emitted program serves both batch modes, the FR-115e hidden-clock rule is enforced **at run time**: the program's vector mode shall refuse at startup — reporting the hierarchical clock refdes and pointing at `--cycles`, exit status 2 — when the design carries a clock generator inside a flattened child or peer sheet, while the free-running mode (FR-117a) drives such clocks time-based normally. (Reworked 2026-07-04; supersedes the original deferred-scope refusal of sub-design instances and off-sheet connectors.) **Switch elements:** a design containing a transmission gate or relay (FR-071g/FR-071h) is refused at generation with a clear "not supported by the fast simulator" message via the tray (FR-074) — switch semantics (FR-083a) are slow-engine-only for now, and refusal is required rather than silent misbehavior; fast-engine support is a possible later additive change. (Added 2026-07-07.) **Deferred scope:** the system never compiles or runs the generated program itself (the user compiles by hand; a server-side toolchain is a possible later change). (Added 2026-07-02.)
- FR-116a: The emitted program shall be split between a fixed, hand-written **C runtime** and thin generated code. The runtime is a checked-in, human-readable pair — **`runtime.h`**, the documented API surface (the four-state value type and combination operators, the net/driver-table structures the generated code populates, the settle/step entry points, and the vector runner), and **`runtime.c`**, its implementation — served to the browser as static assets and copied verbatim beside every generated file (FR-116). Its comments shall be written as API documentation, extractable into user-facing docs later. The runtime re-expresses in one auditable place every semantic the specs pin to the slow simulator: the four states and selectively-pessimistic combination (FR-077/FR-107), net resolution with weak drivers and bus-conflict detection (FR-081–FR-083/FR-108), registered-output edge latching (FR-079/FR-079a), the unit-delay double-buffered step (FR-078/FR-110), the built-in behaviors (clock FR-084, power-on reset FR-071b, input switch FR-087a, pull-up/pull-down FR-083, indicator observation, memory FR-114d), and the batch run semantics (FR-117). The **generated** `<design>.c` contains only design-specific material: the net table and `refdes.pin` label strings (for conflict messages, FR-108), per-instance evaluate functions lowered from the single GALasm compiler's output (FR-109), the driver-contribution wiring, the built-in instance tables with their effective property values (FR-020b), the test-vector column tables (FR-117), and each ROM instance's refdes and content-file path — the contents themselves are read by the program at startup (FR-117b, superseding this FR's original bake-the-bytes rule, 2026-07-03), so the emitted program is self-contained **to compile** while a ROM-bearing design needs its content file at run time, exactly like the editor (FR-114e). (Added 2026-07-02.)
- FR-117: The generated program's first batch mode shall be **test-vector execution**, mirroring §3.19a exactly (this is the fast engine's stimulus and run control; it supersedes the "open" status recorded in FR-110). The generator bakes the design's input and output **column tables** into the program — derived from the design by the same rule as the test-vector panel (FR-115b/FR-115f: input switches, clock generators, indicator bits, and effective-direction ports) — and the program reads **rows** from standard input as plain whitespace-separated text: one line per row, the input symbols (`0`/`1`, plus `C` in a clock column) then a `|` separator then the expected-output symbols (`H`/`L`/`X`), positional against the baked column order. Run semantics are the slow simulator's vector semantics, per FR-107: a combinational design evaluates each row independently, settling to quiescence under the same 10,000-unit bound (FR-115c/FR-085); a clocked design runs the rows in order on persistent state, with the implicit power-on-reset preamble and shared `C` pulses (FR-115e). The program terminates when its input is exhausted, exiting 0 when every row passed and nonzero otherwise. The `.tv` JSON file (FR-115a) is **not** parsed by the C program; its rows are rendered to the plain-text form by tooling outside the emitted program. The **free-running** batch mode — the memory-as-stimulus direction of OQ-012 — is the complementary mode specified by FR-117a. (Added 2026-07-02; free-run mode split out to FR-117a 2026-07-03.)
- FR-117a: The generated program's second batch mode shall be **free-running execution**: invoked as **`--cycles N`** (N a positive integer), mutually exclusive with reading vector rows — when the flag is present, standard input is not read. The built-ins run **time-driven**, mirroring the slow simulator's interactive run rather than the scripted-clock vector semantics: each clock generator emits its FR-084 square wave at its own effective `period` (first rising edge half a period in), and each power-on reset asserts for its FR-071b window (`cycles × clockPeriod` simulated nanoseconds, `clockPeriod` by FR-071b's single-clock rule — the lone clock's effective period, else the 100 ns default). The run advances exactly `N × clockPeriod` unit steps (a design with no clock generator thus runs `N × 100` units), then writes to standard output one line per observable column (the FR-118 set, in column order) of the form `LABEL=v` where `v` is the four-state value `0`/`1`/`U`/`Z`, and exits 0. Standard error carries bus-conflict reports as in vector mode (FR-108/FR-118). The `speed` property (FR-071a) is wall-clock pacing and has no effect in batch. (Added 2026-07-03.)
- FR-117b: The generated program shall read each ROM instance's contents **at startup**, in both batch modes, so a content file can be changed without regenerating or recompiling the program — the fast-engine analogue of the editor's read-at-Run rule (FR-114e). The generator bakes each ROM's **refdes and content-file path** (from its `mem` block, FR-114f) instead of the content bytes (superseding that clause of FR-116a). Per ROM instance, the contents come from: a **`--rom REFDES=FILE`** command-line option naming that instance (repeatable, one per ROM) when given; otherwise the baked path, tried **as recorded** and then by **basename in the current working directory** (so a moved program, or one generated on another machine, finds a file placed beside where it is run). Every load failure is a **hard error** — the program reports to standard error and exits 2: an unresolvable baked path names the ROM's refdes, both paths tried, and describes the `--rom REFDES=FILE` option; a `--rom` naming an unknown or non-ROM refdes, a file with an extension other than `.bin`/`.hex`, or an unreadable or malformed file likewise reports and exits. File format and packing are exactly FR-114e: format by extension (raw binary, or whitespace-separated ASCII hex byte tokens), little-endian `ceil(w/8)` bytes per location, a trailing partial word dropped, content beyond the device's capacity reported to standard error and ignored. Contents are loaded once at startup; the per-row state reset of a combinational vector run re-seeds every ROM from the loaded bytes. RAM devices are unaffected (they still power up all-U). (Added 2026-07-03.)
- FR-118: The generated program shall report on two channels (resolving OQ-011). **Standard output** carries a machine-diffable **transcript** of the vector run: per row, a pass/fail verdict with, for each failing output cell, the actual four-state value observed; then a summary line (rows passed / total) — a stable, line-oriented format suitable for diffing the two engines' results (the FR-107 parity check). **Standard error** carries the bus-conflict reports of FR-108, naming the conflicting drivers in the FR-082 message shape. The **observable point set** is the design's test-vector column set (FR-115b/FR-115f) — indicators, ports, input switches, and clocks. A **`--vcd <file>`** option, available in **both** batch modes (vector, FR-117, and free-running, FR-117a), shall additionally dump a four-state VCD trace of the observable set over simulated time: `$timescale 1ns` (one unit step, FR-110), one VCD signal per observable column, value changes recorded per unit step, mapping 0/1/U/Z onto VCD `0`/`1`/`x`/`z`. (Added 2026-07-02; `--vcd` specified — both modes, timescale, per-step changes — 2026-07-03.)

### 3.24 File Export (Foreign Netlist Formats)

- FR-119: The **File** menu shall include an **Export…** item that writes the current design to a foreign netlist format. Choosing it opens an **export dialog** offering a choice of formats; **NDL** (FR-119a) is the only format initially — the dialog exists so later formats (e.g. KiCad) are additive. Confirming proceeds to the standard save-mode file dialog (FR-047 machinery) with the format's default extension (`.ndl` for NDL) and writes the generated text verbatim through the file-save endpoint (`POST /api/v1/file/save`); cancelling either dialog aborts with no effect. Export is **read-only** — it never modifies the design — and exports the **live in-memory design** (unsaved edits included). Sub-designs and off-sheet connectors are **flattened** first, exactly as Generate C (FR-116, §6.14); a flatten or generation failure refuses the export with a message to the tray (FR-074), and generator warnings post there too. The menu item is disabled while a simulation runs or the test-vector lock is held (consistent with Generate C, FR-116). (Added 2026-07-06.)
- FR-119a: The **NDL backend** shall render the flattened design as an NDL netlist — the plain-text pinout/package/circuit language documented in `docs/netlist-language.md`. Specifically: (a) one `pinout` per distinct component **type** in use, listing every signal pin as `pin <number> = <name>` using the physical pin `number` (FR-062b) — an active-low `/X` pin name exports as `X'` per NDL convention — plus, when the type carries `physical:` metadata (FR-062e), its power pins (under their rail names) and its no-connects (as `NC`); a type with any unnumbered signal pin gets invented lowest-unused numbers and a warning comment in the output; (b) a synthetic **POWER** pinout and package instance carrying one pin per distinct rail name in use, with a rail connection to every part's every power pin — the exported power wiring the logical editor itself does not represent, now derivable from FR-062e; (c) when the design has ports, a **connector** pinout and package instance (`J1`) exposing them — one pin per distinct 1-wide port label, and `<label>0`..`<label>(N-1)` for a multi-bit port (FR-095 naming), numbered sequentially; (d) `package` lines grouping instances by type, with a subunit package's siblings collapsed to their shared U-number stem (`U1A`..`U1D` → `U1`; a flattened child's hierarchical prefix is preserved, `X1/U3A` → `X1/U3`); (e) a `circuit` block named after the design containing, per net (FR-059a semantics, from the live connectivity), one connection statement per sink — a star from the net's **driver** (the first out/tristate/bidir pin, else the first pin) — preceded by the net's name as a comment when it has one; and (f) the virtual built-ins (clock, switch, indicator, indicator8, pull-up/pull-down, power-on reset, and — added 2026-07-07 — the transmission gate and relay, FR-071g/FR-071h), which have no physical package, recorded as **comment lines** naming the instance, its role, and the net pins it drives, observes, or joins (a switch element's comment names its control pin's net and its contact terminals' nets), so no connectivity is silently dropped. Output is **deterministic** for a given design (stable ordering throughout) so successive exports diff cleanly. (Added 2026-07-06.)

## 7. Constraints and Assumptions

**Constraints:**
- The browser application must be implemented in JavaScript (no TypeScript, WebAssembly, or other compile-to-JS languages unless discussed).
- The server must be implemented in Go.
- The system is single-user and localhost-only; no authentication or TLS is required.
- Copy/paste of components is in scope (§3.5a, FR-111–FR-113). (Supersedes the original "out of scope for this phase".)
- Both simulation engines are in scope: the slow debug simulator (§3.19) and the fast engine — a JavaScript, in-browser code generator that emits a standalone C simulator (§3.23, FR-109). (Supersedes the earlier "fast engine out of scope".)

**Assumptions:**
- The user runs a modern desktop browser (Chrome or Firefox); mobile browser support is not required.
- The component-definition file format has been designed and agreed upon (YAML; design.md §7.6) and is capable of expressing all data described in FR-062 through FR-064.
- A single instance of the server runs per user; concurrent multi-session access is not required.
- Design files fit comfortably in memory; no streaming or chunked I/O is needed for save/load.

## 10. Glossary

| Term | Definition |
|---|---|
| Anchor point | A pin on a component where a wire or bus can connect |
| Bend point | A moveable intermediate point on a wire or bus segment that allows the user to shape the route |
| Breakout | Extracting a single bit (signal) from a bus and routing it onward as an ordinary single-bit wire |
| Bus | A multi-bit signal connection rendered as a thick blue line with a width annotation; electrically it is N independent single-bit nets, one per bit |
| Canvas | The drawing surface in the browser on which components and wires are placed |
| GALasm | A logic-equation language used to describe TTL component behavior in YAML files (simulation phase). Evaluated in one of two dialects (FR-066a): *strict* (a `gal:` device named) accepts only that GAL device's language and capacity; *extended* (the default) lifts capacity limits and adds an XOR operator and per-output `.CLK`/`.ARST`/`.APRST` clocks |
| Grid | A uniform lattice of points covering the canvas; all design elements snap to grid intersections |
| Junction | A point at which a wire branches from another wire or bus, electrically tying them together |
| Net | The set of pins and wire/bus segments that are all electrically connected through pins and junctions |
| Palette | The panel displaying available component types as tiles, from which the user selects components to place |
| Pin group | A named set of pins declared in a YAML file that collectively form a bus interface |
| Rat's nest wire | A straight-line wire drawn directly between two pins before the user has routed it with bend points |
| Reference designator | A unique identifier assigned to each component instance (e.g., U1, U2) |
| Rubber-banding | The real-time stretching of wire segments as the user drags a bend point or component |
| Select tool | The default canvas mode in which the user can select, move, and interact with existing elements |
| Slow simulator | The interpretive, in-browser debug simulation engine (§3.19), as opposed to the fast engine (§3.23, a generated standalone C program) |
| Strong / weak driver | A component output is a strong driver of its net; the pull-up/pull-down built-ins are weak drivers, effective only when no strong driver is enabled |
| Unit delay | The slow simulator's timing model: every component's outputs respond exactly one unit (1 simulated ns) after its inputs, regardless of internal complexity |
| Snap-connect | The automatic connection of all pins in a declared pin group when a matching bus is dragged onto a component |
| TTL | Transistor-Transistor Logic; a family of digital logic components (e.g., 74xx series) |
| Wire | A single-bit signal connection rendered as a thin black line |
| YAML file | The component-definition file (`.yaml`; design.md §7.6) defining the name, pin layout, pin directions, pin groups, optional timing, and (eventually) the GALasm behavior of one TTL component type |
| Port | A built-in interface object placed inside a design to expose a net as an external pin; its label and direction define one **one-bit** interface pin (FR-094/FR-095). A multi-bit interface is a portN pin group (FR-071e) |
| Interface | The set of a design's ports — one external pin per distinct 1-wide-port label, or a pin group per multi-bit port — used when the design is embedded (FR-095) |
| Sub-design (hierarchical block) | A separately-saved design embedded in a higher-level design as a single encapsulated component instance (FR-098), rendered as an IC rectangle or a connector strip |
| Off-sheet connector | A port carrying a target file+label that joins peer sheets of one circuit by label and navigates between them on click (FR-101) |
| Flatten | Replacing each sub-design instance with its child's contents (using hierarchical reference designators) so a hierarchical design can be simulated as one circuit (FR-102) |
